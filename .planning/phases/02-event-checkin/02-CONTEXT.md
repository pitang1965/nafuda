# Phase 2: イベント・チェックイン - Context

**Captured:** 2026-04-26
**Source:** ユーザーとの会話（フェーズ再設計）

---

## 設計決定

### 旧設計からの変更理由
当初のプランはチェックアウトを明示的に行う設計だったが、ユーザーから以下のフィードバックあり:
- チェックアウトするまで他画面へ遷移できない印象を与えるUXは不要
- イベントは「主催」と「参加」の2軸で管理したい
- QRコードによる参加フローを中心に据えたい

---

## 確定した設計

### `/events` — イベント一覧ページ（認証必須）
- 2セクション: 「主催したイベント」「参加したイベント」
- 「新規イベントを作成」ボタン

### `/events/new` — イベント作成フォーム（認証必須）
- 入力項目: イベント名・会場名・日付
- 作成後: `/e/$slug` にリダイレクト
- slug はイベント名＋日付から自動生成

### `/e/$slug` — イベントページ（公開）
- イベント情報（名前・会場・日付）
- 参加者一覧（来た人の記録 — チェックアウト概念なし）
- QRコード（このページのURLを表す）
- 「参加する」ボタン（ログイン済みの場合のみ表示 / 未ログインはログインページへ誘導）

### チェックインフロー
1. 主催者がイベントを作成 → `/e/$slug` の QR コードを表示
2. 参加者がQRスキャン → `/e/$slug` に遷移
3. ログイン済みなら「参加する」ボタン表示 → クリックでチェックイン記録
4. 未ログインなら「ログインして参加する」リンク表示

### 参加記録
- チェックインは「来た記録」であり、チェックアウトは不要（UI非表示）
- `checkedOutAt` カラムはDBに残るが機能として廃止
- 参加者一覧は「チェックインした全員」（checkout状態問わず）

### DB変更（追加必要）
- `events` テーブルに `hostUserId: text('host_user_id').notNull()` を追加
  - Better Auth の user.id（主催者判定に使用）
  - `getMyEvents` で `hostUserId = session.user.id` を検索

### 未ログインユーザーの閲覧
- `/e/$slug` は公開アクセス可能
- 参加者のハンドル名・アバターのみ表示
- プロフィールリンクはログイン時のみ有効（OSHI-05 維持）

### QRコード実装
- クライアントサイドのQRコードライブラリを使用
- URL: `window.location.href`（現在の `/e/$slug` ページURL）
- オフライン表示可（Service Worker不要、クライアントレンダリングのみ）

### 廃止するコンポーネント
- `EventCheckinCard`（チェックイン中ステータスカード）は不要
- `checkoutFromEvent` サーバー関数は不要（ただしDBカラムは残す）
- `getActiveCheckin` サーバー関数は不要

### 既存コードの再利用
- `events` テーブル: `hostUserId` 追加マイグレーションが必要
- `event_checkins` テーブル: そのまま再利用
- `checkinToEvent` 関数: 修正して再利用（auto-checkout ロジック削除）
- `getEventParticipants` 関数: checkedOutAt フィルタ削除（全参加者を返す）
