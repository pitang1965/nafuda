# なふだ — ドメイン用語集

このファイルはドメイン語彙の唯一の定義源。実装の詳細や仕様は書かない。

---

## 用語定義

### なふだ
ユーザーが他者に見せるプロフィールカード。アプリ名でもある。コード上は **Persona** と呼ぶ。UI上では常に「なふだ」と表記する（「ペルソナ」はユーザーに見せない）。1ユーザーが複数持てる（例：「婚活用なふだ」「車趣味用なふだ」）。「マイなふだ」は自分のなふだ一覧を指す。

### ペルソナ (Persona)
「なふだ」のコード上の名称。`personas` テーブルに格納。1ユーザーにつき1つが `isDefault = true` のデフォルトペルソナとなり、チェックインの主体になる。コネクションは「つながる」時にユーザーが選んだペルソナが主体になる（複数ペルソナがある場合はピッカーを表示、1つのみの場合は自動選択）。UI上では「なふだ」と呼ぶ。

### ラベル (Label)
なふだを自分だけが識別するための自己識別名。**オーナーにのみ表示され、公開プロフィールには出ない**。「推し活用」「婚活用」「趣味用」など自由記述。コード上は `label`。`personas.label` に格納。任意項目（未設定の場合は displayName のみ表示）。ウィザードでは選んだ[[用途タイプ]]に応じた初期値が入力欄にセットされるが、オーナーが一度でも編集すればその入力が優先され、以後 用途タイプを変えても上書きしない。なふだ作成時に入力欄を用意し、未設定のなふだには設定を促すリンクを表示する。ラベルは"オーナーが呼ぶ名前"であり、"アプリが読む種別"である[[用途タイプ]]とは役割が異なる。

### 用途タイプ (Purpose)
オーナーがなふだに対して選ぶ用途の分類。**アプリがこれを読んで、なふだの見せ方と編集体験を切り替えるスイッチ**として働く。[[ラベル]]（自己識別の自由記述）とは役割が別で、purpose は機械可読な分類であり、オーナーが選択・変更できる。`personas.purpose` に格納（任意）。値は「推し活」「イベント・交流」「車」「その他」などで、特化した用途（例：車）は後から追加していける開いた集合。**null は用途タイプ導入前に作られた既存なふだ**を意味し、従来どおりの表示になる（互換維持）。新規作成するなふだは必ずいずれかの値を持つ（最低でも「その他」）ため、null は既存なふだ専用の状態。purpose の効果はオーナー向け画面（編集体験の出し分け）にとどまらず、公開プロフィールにも波及しうる（タグ見出しの有無・SNSリンクのサジェスト順など）。「その他」は表示上 null と同等だが「明示的に選択済み」という点で意味が異なる。UI上では「用途タイプ」と表記する（この導入の経緯は ADR-0010 を参照）。

### UrlId
ユーザーに1つ割り当てられる英数字識別子。**システムが生成するランダム文字列**であり、ユーザーは選択できない。変更不可。`url_ids` テーブルに格納。`/u/{urlId}` 単独のルートは存在しない。すべてのなふだへのアクセスは ShareToken を伴う URL 経由のみ。UrlId 自体はアクセス制御に関与しない（推測耐性は ShareToken が単独で担う）。

### ShareToken
ペルソナごとに発行されるランダムな16進数トークン。QRコードのURLに埋め込まれ、プロフィール参照に使われる。デフォルトペルソナも含め、すべてのペルソナが固有の ShareToken を持つ。`personas.share_token` に格納。

### プロフィールQR
`/u/{urlId}/p/{shareToken}` 形式の永続URLをエンコードした画像。「なふだを見せる」で使用する。インターネット公開・印刷も可能。スキャンしてもコネクションは生成されない。

### つながりQR
「なふだを交換する」で生成される短命（15分有効）のQRコード画像。スキャンした時点でサーバーに保留中の招待が記録される。コネクションはスキャン者が「つながる」をタップした時点で双方向・即時に成立する。

### イベント (Event)
参加者がチェックインできる**開催期間（いつからいつまで）と場所のある記録単位**。場所は**物理会場でもオンライン（Zoom 等）でもよい**——online/offline をイベント種別として区別はせず、「どこで」が会場名でもオンラインの場であってもよいだけ。開催期間は単一の日付ではなく `[開始, 終了]` の範囲で持つ（日付のみの範囲＝複数日開催も、日時の範囲＝当日の時間帯も表せる）。チェックインの可否はこの開催期間で律する（[[チェックイン]]参照）。2種類ある：

**企画イベント** — 誰かが事前に作成した催し（ライブ・コンサート・ファンミ・オフ会・勉強会等）。`/events/new` から明示的に作成する。会場名（`venueName`）必須。公開イベントページ（`/e/{shareToken}`）を持つ。

**即時イベント** — 「なふだを交換する」フロー中にその場で作成する出会いの文脈記録（「バーナイト＠立川」等）。`events.isInstant = true`。イベント名と GPS 座標のみで作成可能（会場名は任意）。スラッグは自動生成。作成と同時に作成者がチェックインした状態になる。コネクションが成立しなくても作成される。**公開ページを持たない**（他者からアクセス不可）。作成者のイベント一覧には企画イベントとは別セクションで表示される。コネクション相手のイベント一覧には表示されない（コネクションの文脈テキストとしてのみ参照可能）。

両種別とも `events` テーブルに格納。作成時にどのペルソナで作ったかを `hostPersonaId` として記録する。`events.gpsCoordinates`（任意）に GPS 座標（緯度・経度）を保存する。

**イベントに含まれないもの：**
- 個人の聖地巡礼・撮影地訪問（個人の行動であり、主催者のいる催しではない）
- 趣向・コミュニティ（「○○ファン」「車中泊グループ」等）

聖地を使った企画催し（「○月○日に聖地○○に集まれ！」等）は主催者が作成したイベントとして扱う。個人の聖地訪問を記録したい場合は推しタグで代替する。

### チェックイン (Checkin)
ペルソナがイベントに「入る」アクション。`checkedOutAt IS NULL` の状態がアクティブ。新しいチェックイン時に直前のチェックアウトが自動記録される（同時に複数イベントへの参加不可）。複数のなふだを持つユーザーにはペルソナピッカーを表示し、どのなふだとして参加するかを選ぶ（1つのみの場合は自動選択）。同一ユーザーが同一イベントに別のなふだで再チェックインした場合、以前のなふだのチェックイン記録は削除される（1ユーザー = 1エントリ）。**チェックインは[[イベント|開催期間]]内（前後に短い猶予を含む）に限られる**——範囲外（後日 URL を拾った等）からの参加は受け付けない。意味するところは「**開催中のイベント（物理会場でもオンラインでも）にその時居た**」であり、QR/URL の提示自体が"その場に居る人だけ"のゲートを兼ねる（GPS 近接判定は持たない→ ADR-0020）。企画イベントでは開催**終了**をもってアクティブなチェックインが自動的にチェックアウトされる（開きっぱなしの古いアクティブ状態が後続のコネクション文脈を誤らせないため／[[コネクション]]①）。即時イベントは開催期間を持たないが、その文脈は**交換のその場でのみコネクションにコピーされ、残存するアクティブ状態を後続の交換で再利用しない**ため、チェックインの寿命管理を必要としない（→ ADR-0020）。参加者一覧には開催期間中に参加を表明したペルソナを表示する。`event_checkins` テーブルに格納。

### コネクション (Connection)
2つのペルソナ間の出会いの記録。生成経路は**なふだを交換する**（つながりQR経由）のみ。アカウント未所持の相手は[[保留中の招待]]を介して後からこの同じ接続を完成させる（出会い自体は対面のQRスキャンであり、別経路ではない）。A→BとB→Aの2行が同時に生成される（即時双方向）。**イベント参加者一覧からのオンライン接続（旧「つながりリクエスト」）は設計から削除した**——チェックインが実在証明になっておらず、会っていない相手からの申請というスパム構造を呼び戻すため（→ ADR-0013）。削除は非対称で、各自が自分の行を独立して削除できる（相手の行には影響しない）。複数のなふだを持つユーザーには接続時にペルソナピッカーを表示する（1つのみの場合は自動選択）。各コネクションには「出会いの文脈」（いつ・どこで）が付与される。文脈の付与経路は以下の通り：

- **①自動（企画イベントチェックイン）** — QR を**発行した側**（A）が**企画イベント**にチェックイン中の場合、そのイベント情報（`eventId` 付き）が双方のコネクション行に自動付与され、**編集ロック**される。企画イベントは公開ページを持つ客観的な共有事実なので、双方の記録で同一・固定とする。
- **②自動（即時イベント作成）** — 「なふだを交換する」タップ後、QR 表示前に「今どこで？」スクリーン（スキップ可）が表示され、イベント名を入力すると即時イベントが作成されてAがチェックインする。GPS 座標は A の端末から自動取得（位置情報許可が必要）し `events.gpsCoordinates` に保存。**即時イベントの文脈は A と B 双方のコネクション行に `eventId` 付きでコピーされる**。即時イベントは**交換ごとに新規・独立**に作られる（同じ場で複数人と交換しても1対1ごとに別の即時イベントになり、束ねない——「誰が参加したか」のグルーピングは企画イベントの名簿が担う）。ただし企画イベントと違い**編集ロックしない**——即時イベントは公開ページを持たない A のその場の私的ラベルなので、双方とも③の後付け編集で自分の言葉に書き換えられる。B の初期値は A が書いた文脈だが、**コピーは成立時点のスナップショットであり、以後 A・B 各自の編集は相手の行に伝播しない**（各行は持ち主だけが見る・編集する独立した記録）。編集後はコネクション側の名前が `events.name`（正本）と乖離しうるが、これは意図的な区別である（`eventId`＝出会いの正本イベント／コネクション側の名前＝各自の私的ラベル）。スキップした場合は文脈なしで QR を表示する。
- **③後付け編集** — **企画イベント由来（`eventId` 付き かつ `isInstant = false`）の文脈のみ編集不可**（公開ページを持つ客観的事実として固定）。それ以外——`eventId` を持たないコネクション（文脈が空）、または即時イベント由来（`isInstant = true`）——は、コネクション成立後に場所名・イベント名をフリーテキストで書き足し・再編集できる。編集はコネクション側の非正規化コピーにのみ書き込み、`events` 本体や相手の行には影響しない。
- **④プライベートメモ** — 自分だけが見られる相手についての覚え書き（`connections.privateMemo`）。コネクション成立後に追記・編集できる。

なふだ交換時の既存イベント選択は設計上行わない（誤選択リスクのため）。文脈を提供するのは常に**つながりQR を差し出す側（発行者A）**で、B はそれを受け取る。A が**企画イベント**にチェックイン中（＝開催期間内のアクティブ状態）であれば、その情報が自動的に文脈として使われる（①）。**即時イベント由来の文脈は交換のその場で確定するもので、過去の残存アクティブチェックインからは引かない**（古い即時チェックインが後続の交換を汚さないため／→ ADR-0020）。コネクション一覧の「どのなふだとして」表示には、接続時点のラベルではなく**現在のラベル**を使う（ペルソナテーブルをJOINして取得）。ラベルを変更すると過去のコネクション表示も更新される。`connections` テーブルに格納。

### なふだを見せる
自分のプロフィールQRを他者に提示するアクション。永続URLのためインターネット公開・印刷もできる。閲覧が目的であり、コネクションは生成されない。

### なふだを交換する
つながりQRを生成・提示し、相手がスキャンして「つながる」をタップするアクション。紙の名刺交換に相当する。コネクションが双方向・即時に成立する。

**フロー：**
1. 「なふだを交換する」をタップ
2. 「今どこで？」スクリーンが表示される（スキップ可）。イベント名を入力すると即時イベントが作成されてチェックイン状態になる。GPS 座標は自動取得。スキップすると文脈なし。
3. QR（つながりQR）が表示される。
4. 相手がスキャンして「つながる」を押すとコネクション成立。A 側には「つながりました！」通知。
5. 「**交換しない**」を選ぶとつながりQRトークンが即時削除される。「**交換予定**」または通知後の「**閉じる**」を選ぶとトークンはそのまま（15分有効）でシートを閉じる。

### 保留中の招待 (Pending Invite)
つながりQRをスキャンした時点でサーバーが作成する一時的なDBレコードで、**スキャンした人が48時間以内に接続を完成できる、QRトークンの寿命に依存しない恒久的な接続権**。つながりQR URL（15分有効）とは独立して存在するため、URLが期限切れになっても有効。スキャナーのブラウザ（localStorage の inviteToken）に束縛され、別端末には引き継がない。**ログイン有無を問わず作成する**——アカウント未所持の人がアカウントを作る猶予になるだけでなく、ログイン済みでもページを開いたまま15分を超過するとQRトークンが切れるため、後追い接続の受け皿として全スキャナーに必要（当初はアカウント未所持専用だったが、ログイン済みでも交換し損ねるケースを救えなかったため対象を広げた）。アカウント未所持の場合はアカウント・ペルソナ作成後に自動適用され、コネクションが双方向・即時に成立する。文脈は招待作成時の発行者アクティブチェックインからスナップショットし、適用時に取り直さない（[[コネクション]]／ADR-0012）。

### 同担拒否 (Dojin Reject)
同じ推しを応援するファン（同担）との交流を望まないというユーザーの意思表示。`personas.dojin_reject = true` の場合、そのペルソナはイベント参加者一覧から非表示になる。

### 推しタグ (Oshi Tags)
ペルソナに紐づく自由記述のタグ配列。応援しているアーティスト・キャラクター・作品などを登録する。任意項目（未設定のまま使用できる）。`personas.oshi_tags` に格納。推し活は[[打ち出し]]上「最も濃い実例」として扱われるユースケースだが、表の顔ではない（→ ADR-0009）。「推しタグ」はコード/データ上の名称であり、UI上の見出しは[[用途タイプ]]に応じて変わる（`oshi`＝「推しタグ」／非 `oshi`＝「興味タグ」／`null`＝従来表示の「推し / 趣味タグ」）。非推し活のなふだでは同じフィールドが「興味タグ」として提示される（"コード名 vs UI名"の関係は[[なふだ]]／Persona と同じ）。

### 打ち出し (Positioning)
アプリがユーザーに見せる「表の顔」＝ヒーロー・第一印象・SEOの主軸。**「その場で会った“仕事じゃない”人と、SNSでつながる」**を核に据える。対象は名刺を持たない／出さない場での出会い（勉強会・趣味のオフ会・懇親会やパーティー・バーや居酒屋での出会い等）。**ビジネス交流会は対象外**（紙名刺＋Eight の縄張りであり戦場にしない）。差別化は「Eight＝仕事の名刺／なふだ＝“仕事じゃない”出会いの名刺」。推し活・趣味・コミュニティ等の汎用性は**プロダクトの射程と二次メッセージに残す**が、SEOの核には据えない。推し活は最も拡散する実例コンテンツとして一等地に配置する。この打ち出しへの転換の経緯は ADR-0009 を参照。

### 退会 (Account Deletion)
ユーザーがアカウントとすべての関連データを即時・完全に削除するアクション。UI上では「退会する」と表記。削除対象はユーザーレコード・すべてのペルソナ・SNSリンク・[[なふだリンク]]・[[ギャラリー]]写真（ストレージ実体も含む）・チェックイン履歴・コネクション（相手側の記録も含む）・UrlId。退会後に同じメールアドレスで再登録は可能だが、新しい UrlId が発行されるため以前の QR コードは無効になる。退会してもそのユーザーが作成したイベントは削除されない（他の参加者の文脈として機能するため）。

### なふだを削除する (Persona Deletion)
オーナーが自分のなふだ（ペルソナ）を1枚だけ削除するアクション。[[退会]]（アカウントごと全削除）とは別概念で、UrlId・アカウント・他のなふだは残る。**2枚以上あるうちの1枚を消す操作に限定**し、最後の1枚は削除できない（全削除は退会に委ねる）。なふだの編集画面から確認付きで実行する。削除すると、そのなふだに紐づくSNSリンク・[[なふだリンク]]（このなふだが持つもの＋**他のなふだからこのなふだを指していた被参照**の双方）・[[ギャラリー]]写真（ストレージ実体も含む）・チェックイン履歴・つながりQRトークン・コネクション（相手側の記録も含む）が消える。削除対象が[[ペルソナ|デフォルトペルソナ]]だった場合は残るなふだのいずれかが新しいデフォルトになる。

### SNSリンク (SNS Link)
ペルソナに紐づく**外部サービス**へのリンク。プラットフォーム種別・URL・表示順を持つ。`sns_links` テーブルに格納。対応プラットフォーム: x, instagram, tiktok, youtube, discord, line_openchat, github, spotify, facebook, minkara, linkedin, note, pixiv, other。**`nafuda.me` ドメイン（およびそのサブドメイン）の URL は SNSリンクには登録できない**——SNSリンクは定義上「外部サービス」へのリンクであり、自社ドメインは対象外。プロフィールURL（見せる/つながる）・イベントURL を含むあらゆる `nafuda.me` URL が対象。自分の別のなふだへの導線は[[なふだリンク]]が担い、他人の[[ShareToken]]を勝手に公開してしまうのも防ぐ。

### なふだリンク (Nafuda Link)
オーナーが自分のなふだに貼る、**自分が持つ別のなふだ**への内部参照リンク。外部サービスを指す[[SNSリンク]]とは別概念。リンク先は URL 文字列ではなく `targetPersonaId` で参照し、表示名・アバターは参照先[[ペルソナ]]から動的に取得する（リンク先の表示名を変えると各なふだリンクの表示も更新される）。**他人のなふだは指せない**——自分の `userId` のペルソナのみをピッカーで選ぶため、所有権が構造的に保証される。公開プロフィールでは「なふだアイコン＋なふだ名」のようなコンパクトなチップ（アバター丸＋displayName、アバター無しはなふだアイコンにフォールバック）で横並び表示する。リンク元・リンク先どちらのなふだが削除されても残らない（双方向カスケード）。`nafuda_links` テーブルに格納し、表示順を持つ。UI上では「なふだリンク」と表記する（この設計の経緯は ADR-0015 を参照）。

### 公開範囲 (Field Visibility)
ペルソナのフィールドごとの公開/非公開設定。`personas.field_visibility` に JSON で格納。デフォルトは全フィールド公開（QRを渡す行為自体が公開の意思表示という前提）。対象フィールドは SNSリンク・推しタグ・アバター・[[ギャラリー]]。ギャラリーは写真1枚ごとではなく**ギャラリー全体で1トグル**（`gallery` キー）として切り替える。

### ギャラリー (Gallery)
なふだに載せる、アバター（プロフィール画像）以外の写真の集まり。オーナーが「対象物」を画で見せるためのコンテンツで、推し・作品・愛車などを写真で伝える（動機の出発点は「Linktreeは写真を出せないのが残念」という実ユーザーの声）。[[なふだスタイル]]（コード管理の装飾パッケージ）とは別レイヤーで、これは**ユーザーがアップロードするコンテンツ**。何にも紐づかない独立した1セットで、[[推しタグ]]やSNSリンクとは無関係（将来の紐づけ拡張は妨げない）。各写真は任意の短いキャプション（空可）を持てる。**最大6枚**。公開プロフィールでは正方形サムネイルのグリッドで表示し、タップで全体表示（ライトボックス）する。[[公開範囲]]の対象（`gallery` トグル）。`gallery_photos` テーブルに格納し、表示順を持つ。アバターと同じく差し替え・[[なふだを削除する|なふだ削除]]・[[退会]]時にストレージから物理削除する（孤児を残さない）。UI上では「ギャラリー」と表記する。

### なふだスタイル (NafudaStyle)
なふだ（ペルソナ）の見た目を決める装飾パッケージ。背景（CSSグラデーション／単色）・フォント（Google Fonts）・縁取り（SVGデコラティブフレーム、なしもあり）の組み合わせをひとつのパッケージとして提供する。ユーザーはパッケージをまるごと選択する（個別要素のカスタマイズは不可）。コード上は文字列IDで識別し（例: `'sakura'`）、`personas.style_id` に格納する。IDの実体（CSS定義・SVGアセット）はコード側で管理する。将来的に有料パッケージを追加できるよう、スタイル定義に `isFree` フラグを持つ。現時点では全スタイルが無料（`isFree: true`）。課金ゲートを実装する際にこのフラグを使って選択制限を追加する。一部のスタイルはホログラフィック効果（`holographic: true`）を持ち、端末の傾き（DeviceOrientation API）に連動してカード上で虹色の光沢が動く。UI上では「なふだスタイル」と表記する。適用範囲は公開プロフィールページ（`/u/{urlId}/p/{token}`）。なふだスタイルは**コード管理の装飾**であり、ユーザーがアップロードする画像（背景画像など）は持たない——ユーザーが載せる写真は装飾ではなくコンテンツとして[[ギャラリー]]が担う（この境界の経緯は ADR-0006・ADR-0014 を参照）。

## 重要な区別

| 混同しやすいペア | 区別 |
|---|---|
| なふだ / ペルソナ | 同じ概念。UIでは「なふだ」、コードでは「Persona」と呼ぶ。ユーザーに「ペルソナ」という語は見せない |
| UrlId / ShareToken | UrlIdはユーザー単位・変更不可の識別子で、URL構造上の名前空間（`/u/{urlId}/...`）を与えるのみ。アクセス制御は ShareToken（ペルソナ単位・QR用トークン）が単独で担う |
| プロフィールQR / つながりQR | プロフィールQRは永続・閲覧専用。つながりQRは15分有効・コネクション生成可能。同じなふだに対して別々に存在する |
| チェックイン / コネクション | チェックインは「開催中のイベントにその時居る」状態（物理会場でもオンラインでも）。コネクションは「つながった」という記録 |
| なふだを見せる / なふだを交換する | 「見せる」は閲覧のみ・コネクション不発生。「交換する」は短命QRを使ったその場の名刺交換でコネクションが成立する |
| displayName / ラベル | displayNameは他者に見せる公開名。ラベルはオーナーだけが見る自己識別用の名前（最大20文字）。コネクション一覧など自分向けの画面ではラベルを優先表示し、未設定の場合は displayName にフォールバック |
| ラベル / 用途タイプ | ラベルはオーナーが自由記述する"自分が呼ぶ名前"（非公開）。用途タイプはオーナーが選ぶ"アプリが読む種別"で、見せ方・編集体験を切り替えるスイッチ（効果は公開ページにも波及しうる）。両方とも「用途」に関わるが、ラベルは識別のための任意の文字列、用途タイプはアプリが分岐に使う有限の分類 |
| なふだスタイル / テーマ | なふだスタイルはこのアプリの用語。「テーマ」「スキン」は使わない |
| ギャラリー / アバター | どちらも写真だが、アバターは「自分（顔）」を表す1枚のプロフィール画像、ギャラリーは「対象物（推し・作品・愛車）」を見せる最大6枚のコンテンツ。役割が別 |
| ギャラリー / なふだスタイル | ギャラリーはユーザーがアップロードするコンテンツ（写真）。なふだスタイルはコード管理の装飾パッケージ。ユーザー画像は装飾には持ち込まない（背景画像のカスタマイズはしない）。レイヤーが別 |
| 企画イベント / 即時イベント | 企画イベントは事前作成・公開ページあり・参加者が能動的にチェックイン。即時イベントは「なふだを交換する」フロー中に作成・公開ページなし・作成者のみ一覧に表示 |
| SNSリンク / なふだリンク | SNSリンクは外部サービスへのリンク（URL文字列を保存）。なふだリンクは自分の別のなふだへの内部参照（`targetPersonaId` で参照、他人のなふだは不可）。`nafuda.me` ドメインのURLはSNSリンクには登録できない（内部ドメインは外部サービスではない） |
| なふだを削除する / 退会 | なふだの削除は1枚だけを消す操作（他のなふだ・UrlId・アカウントは残る／最後の1枚は不可）。退会はアカウントごとすべてのなふだ・UrlIdを完全削除する。両者とも関連エンティティ（コネクション・チェックイン・SNSリンク）のカスケード削除を伴うが、退会のみ UrlId とユーザーレコードを消す |
| 自分から見た画面 / 人から見た画面 | 「自分から見た画面」はオーナー本人が見る自分のなふだ（マイなふだ／編集）で、**非公開項目もオーナーには見える**（🔒付きで表示）。「人から見た画面」は[[公開範囲]]を適用して**非公開項目を完全に除いた**他者視点の表示で、これが[[プロフィールQR]]（[[なふだを見せる]]で渡る永続URL）が表示する公開ページの姿。両者は同じなふだでも見える内容が異なる（公開プロフィールはビューワー視点で固定＝viewer-agnostic、ADR-0019）。「人から見た画面の確認」という要望はこの後者を指す |
