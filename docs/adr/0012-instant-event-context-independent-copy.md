# ADR-0012: 即時イベント文脈は双方のコネクションに独立コピーする

- **ステータス**: 採用
- **決定日**: 2026-06-14

## 背景

「なふだを交換する」フロー中に作成される即時イベント（「今どこで？」で入力する出会いの文脈）は、当初 **A（QR を発行した側）のコネクション行にのみ** 文脈を付与し、B（スキャンした側）の行には付与しない設計だった（旧 CONTEXT.md ②）。理由は「B はスキャン時に A が書いたメモを見ていないため、B は自分の言葉で後付け（③）すべき」という認識論的な配慮だった。

しかしこれは、つながり一覧で経路ではなく当事者によって表示が変わる非対称を生む。同じ1回の交換なのに、A の画面にはイベント名が出るが B の画面は「誰として・いつ」だけになる。B 側は空欄スタートで、③の導線も弱く、実際には誰も書かない。

加えて、コネクションは ADR-0007 で **A→B と B→A の独立した2行**として保持しており、削除も各自が自分の行を独立して行う（非対称削除）。文脈もこの「各行は持ち主のもの」という思想に揃えるのが自然である。

## 決定

即時イベントの文脈（イベント名・場所名・日時）を、**A と B 双方のコネクション行にテキストとしてコピーする**。ただし `connections.eventId` は **両方とも null** とする。

- B の初期値は A が「今どこで？」に書いた文脈。ただしコピーは **成立時点のスナップショット**であり、以後 A・B 各自の編集は相手の行に伝播しない（共有元を持たない独立した記録）。
- `eventId` が null なので、両行とも既存の③後付け編集（フリーテキスト）にそのまま乗り、各自が自分の言葉に書き換えられる。
- 企画イベント由来の文脈は従来どおり双方の行に `eventId` 付きで付与し、編集ロックする（①）。即時イベントとの差はこの `eventId` の有無に集約される。

`contextLocked`（`eventId !== null`）の判定ロジックは変更しない。即時行は `eventId` が null になることで自然に編集可能になる。

## 却下した代替案

- **現状維持（B には付与しない）**: 指摘どおりの非対称が残る。B の画面が空欄のままになる痛点を解消できない。
- **対称ロック（企画と同じ扱い）**: 双方に `eventId` 付きでロックコピーする。実装は単純化（種別分岐が消える）するが、即時イベント名は公開ページを持たない A のその場の私的ラベルであり、それを B の記録に客観的事実として固定するのは強すぎる。各自が自分の記憶として直せない。
- **共有元 + 伝播（ライブ JOIN / 編集時再同期）**: 即時イベントを A 所有のミニ企画イベントとして扱い、A の編集を B へ伝播させる。新たな編集 UI・伝播機構（JOIN 化または再同期）が必要で、ADR-0004 のラベル JOIN とも状況が異なる（イベントは削除時に非正規化コピーを凍結記録として残す `onDelete: set null` 設計のため、JOIN 化はフォールバックの再設計を伴う）。得るもの（自動伝播）に対し触る面が大きい。

## トレードオフ

- **失うもの**: 即時イベント文脈の「事実としての固定」。A が「今どこで？」に書いた内容も A 自身が後から書き換えられる（ただし A 自身の入力なので改善と捉える）。また A・B のコピーが編集により乖離しうる。
- **得るもの**: つながり一覧の当事者間非対称の解消。ADR-0007 の「各行は持ち主が独立して持つ」思想との一貫性。新規 UI・伝播・JOIN を伴わない最小実装。「事実（企画イベント）vs メモ（即時イベント）」の区別を `eventId` の有無で明快に表現できる。
- **許容できる理由**: 即時イベント文脈は公開ページを持たない私的な覚え書きであり、A・B それぞれが自分の記憶として保持・編集できることに価値がある。両者にしか見えず、不快なら削除もできる（ADR-0007 の非対称削除）。乖離は「各自の記録」として許容される。
