# ADR-0005: 退会時のデータ削除ポリシー

- **ステータス**: 採用
- **決定日**: 2026-06-01

## 背景

退会機能（アカウント削除）の実装にあたり、ユーザーに紐づく複数のエンティティ（ペルソナ・コネクション・イベントなど）をどう扱うかを決定する必要があった。

## 問題

ユーザーを単純に削除するだけでは不十分で、関連エンティティの扱いが複数の方向に分岐する。

- **コネクション**: 相手ユーザーの記録にも自分のペルソナへの参照が残る
- **イベント**: 自分が作成したイベントに他のユーザーがチェックインしている可能性がある
- **UrlId**: 既存のQRコードがそのURLを指している

## 決定

### 即時ハード削除

猶予期間なし・完全削除を採用する。削除完了後、ユーザーはログアウト状態でログイン画面へ遷移する。

### 削除カスケード（削除するもの）

| エンティティ | 処理 |
|---|---|
| `session` | 全セッション削除（即時ログアウト） |
| `account` | OAuthアカウントリンク削除 |
| `connections` | `fromPersonaId` または `toPersonaId` が自分のペルソナを指すレコードをすべて削除（相手側の記録も含む） |
| `event_checkins` | 自分のペルソナによるチェックイン記録を削除 |
| `sns_links` | 自分のペルソナに紐づくSNSリンクを削除 |
| `personas` | 全ペルソナを削除 |
| `url_ids` | UrlIdレコードを削除 |
| `user` | BetterAuthユーザーレコードを削除 |

### 保持するもの（削除しないもの）

| エンティティ | 処理 | 理由 |
|---|---|---|
| `events` | `hostUserId` を NULL に更新して保持 | イベントは他の参加者の文脈として機能する。削除すると無関係なユーザーの履歴が破壊される |

### 再登録

退会後、同じメールアドレス・OAuthアカウントでの再登録を許可する。ただし新しい UrlId が発行されるため、以前のQRコードは無効になる。

## 却下した代替案

- **ソフト削除（論理削除・猶予期間あり）**: 削除済みユーザーのセッション管理・コネクション表示の扱いが複雑になる割に、誤退会ユースケースが希少なため却下
- **コネクションをゴースト表示で残す**: 「退会済みユーザー」として表示するには削除前の情報を非正規化して保持する設計が必要になり複雑さが増す。また「存在しない人とつながっている」という表示はUXとして不自然なため却下
- **イベントを削除する**: 退会1人の行動が他ユーザーの履歴を壊すため却下。`hostUserId` はスキーマ上すでにNULLableであり、NULL更新で自然に対応できる

## トレードオフ

- **失うもの**: 誤退会の取消手段（即時削除のため復元不可）
- **得るもの**: 「削除した」という感覚が明確。個人情報保護法上の削除要求への対応がシンプル
- **許容できる理由**: 確認モーダル＋チェックボックスによる摩擦で誤操作を十分に防げる
