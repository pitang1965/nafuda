# ADR-0018: ボトムナビ常設のアプリシェル導入

- **ステータス**: 採用
- **決定日**: 2026-06-20

## 背景

保護画面（`_protected` 配下）のヘッダーは各画面が個別に手書きしていた。ホーム（`/me`）は右上にバーガーメニュー（`AppMenu`）＋なふだ切り替え＋編集、サブ画面（つながり・アカウント・イベント）は「戻る＋タイトル」を各自で組んでおり、フォントサイズ（`text-lg`/`text-base`/`text-xl`）・border の有無・戻る矢印SVGの重複など、見た目と実装がバラついていた。`_protected.tsx` は `<Outlet />` のみで共有レイアウトを持たない。

ナビゲーションの常設も課題だった。`AppMenu`（バーガー）はホームにしか無く、サブ画面では戻る連打でしか主要画面に出られない。主要な行き先（なふだ・つながり・イベント）への横移動導線が存在しなかった。

## 問題

「ヘッダー/フッターは固定したまま `main` だけ切り替わる」アプリシェルを導入したい。その際に解くべき分岐が2つあった：

1. **ホームの全面没入スタイルとの両立** — `/me` は「なふだスタイル」連動の全面背景を持ち、署名的なカード体験を成す。中立的な常設 chrome を被せるとカードが箱詰めされ、没入感が失われる。
2. **下部バーの役割** — 「なふだ切り替え・編集」はホーム固有の操作であり、どの画面でも同じ常設バーには載せられない（他画面で無意味になる）。下部に置けるのは本質的にグローバルな行き先だけ。

## 決定

**モバイル/PWA の定石（iOS Tab Bar・Material 3 Bottom Navigation）に倣い、`_protected` にボトムナビ常設のアプリシェルを導入する。** 各論は以下。

### 1. 常設ヘッダー（細）
画面タイトル＋右上バーガー（`AppMenu`：アカウント・プライバシーポリシー・ログアウト）。アカウント類の二次導線はここに集約し、独立した情報フッターは作らない（プライバシーリンクもバーガー配下に集約）。

### 2. 常設ボトムナビ（3タブ）
**なふだ（ホーム `/me`）／つながり（`/connections`）／イベント（`/events`）** の3つを主要な行き先としてボトムナビに置く。アカウント/設定/法務などの二次的項目はタブに昇格させず、右上バーガーに残す（主要な行き先＝下、アカウント類＝メニュー、の役割分離）。

### 3. ボトムナビは原則として全保護画面で常時表示、集中フローのみオプトアウト
深いプッシュ画面（`events/new` 等）でもボトムナビを常に出すのを既定とする。例外として、**自前の多段ナビや未保存変更ガードを持つ集中フロー**は `staticData.hideChrome` でシェル（ヘッダー/ボトムナビ）ごと外す。具体的には初回オンボーディング `profile/wizard`（独自の段階ナビ）と `profile/edit`（`anyDirty` による「保存されていない変更があります」確認）が該当。後者はシェルのタブ/戻るがこのガードを素通りして編集内容を失わせるため、自前の戻る付きで全画面化する。出し分けはルートが宣言する `hideChrome` フラグ1つで足り、シェルに複雑な分岐を持ち込まない。

### 4. ホームはシェルに入れ、スタイルは main 内に限定
`/me` もシェル配下に置く。ヘッダー/ボトムナビは**中立色**（スタイル非追従）とし、なふだスタイルの背景は**中立 chrome に挟まれた `main` 内だけ**を塗る。全面没入は手放す。ホーム固有の操作（なふだ切り替え・編集・カード・「なふだを見せる」「なふだを交換する」）は `/me` の `main` 内に維持し、上部バーガーのみヘッダーへ移す。

### 5. 「つながりを見る」ボタンを撤去
`/me` 内の `/connections` への導線ボタンは、つながりタブと重複するため撤去する。署名アクション「なふだを見せる」「なふだを交換する」はコンテンツとして残す（タブは“行き先”に限定し、アクションを混ぜない）。

### 6. 公開ページは対象外
`/u/{urlId}/p/{token}`・`/e/{slug}`・`login`・`privacy`・`connect`・トップ `index` は `_protected` の外であり、シェルを適用しない。特にスタイル付き公開プロフィールにアプリ chrome は載せない。

## 検討した代替案

- **バーガーのみ継続（ボトムナビ無し）**: 実装は最小だが、主要な行き先への横移動が常に2タップ以上（メニューを開いてから選択）で、最頻導線を親指の届く下部に置けない。定石から外れる。
- **下部を画面別アクションバーにする**: ホームでは「切り替え・編集・交換」、他画面では各画面のアクションを下部に出す案。親指 ergonomics は満たすが、画面ごとに中身が変わるため「常設（どこでも同じ）」にならず、`main` だけ swap するシェルの純度が崩れる。グローバルな行き先（なふだ/つながり/イベント）の発見性も上がらない。
- **メニューも4つ目のタブにする**: ナビを全部下部へ集約できるが、アカウント/設定は最頻の行き先ではなく、主要3タブの粒度を薄める。役割分離（行き先＝下／アカウント＝メニュー）を優先して見送り。
- **ホームを没入のままシェル対象外にする**: カードの全面没入を保てるが、ホーム⇔サブの境界でヘッダー/ボトムナビが再マウントされ「常設」が崩れる。「没入はそれほど必要ない」との判断に基づき、ホームもシェルに入れる方を採った。
- **chrome をスタイル追従・透過にしてホームでも背景の上に乗せる**: 没入を保ちつつ常設も両立できるが、スタイルごとに前景色を出し分ける実装コストとデザイン複雑度が高い。

## 結果・トレードオフ

- 主要3画面への横移動が全画面から1タップになり、ナビゲーションが常設化される。各画面の手書きヘッダーがシェルへ集約され、見た目と実装のバラつきが解消する。
- `/me` の全面没入スタイルは失われ、なふだスタイルは `main` 内に箱詰めされる。プロダクトの署名的体験を一段弱める代償を受け入れた。
- ボトムナビを深い画面でも常時表示するため、`events/new` のような未保存ガードを持たないフォームでは、タブ誤タップで入力を失うリスクが残る（元から戻るで失われていたため悪化はしない）。問題が顕在化したら当該ルートにも `hideChrome` を広げる。
- サブ画面（つながり・イベント等）はタブのルートとして到達した場合は戻るボタンを持たない。タイトルはシェルヘッダーが表示する。各画面の「戻る＋タイトル」上部バーはシェル導入時にリファクタリングが必要。
- 「アプリシェル」「ボトムナビ」「共通ヘッダー」は UI 実装の語彙でありドメイン概念ではないため、`CONTEXT.md`（ドメイン用語集）には登録しない。
