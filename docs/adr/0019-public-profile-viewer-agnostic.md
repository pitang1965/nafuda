# ADR-0019: 公開プロフィールは閲覧者非依存（ログイン状態でも見せ方を変えない）

- **ステータス**: 採用
- **決定日**: 2026-06-21

## 背景

ADR-0018 でボトムナビ常設のアプリシェル（`_protected` 配下）を導入した。その結果、`/me` の「📛 他のなふだ」チップ（[[なふだリンク]]）からシェル外の公開プロフィール `/u/{urlId}/p/{token}` へ遷移すると、ヘッダーもボトムナビも「← 戻る」も無い状態に落ちる、という指摘が出た。

ここで2つの問いが立った：

1. **オーナーが自分の `/me` から「他のなふだ」をタップしたとき** — 公開ページへ飛ばすべきか、それとも上部のなふだ切り替え（PersonaSwitcher）と同じ in-place 切り替えにすべきか。
2. **オーナー以外（特にログイン済みの別ユーザー）が公開ページを開いたとき** — ログイン状態に応じてアプリシェルを被せる等、見せ方を変えるべきか。

## 問題

公開プロフィール `/u/{urlId}/p/{token}` は「[[なふだを見せる]]」の正本ビューで、QR スキャン・共有リンクから**誰でも開ける閲覧専用ページ**である。`head()` に OGP/Twitter メタを持ち、クロール・シェア前提で、認証セッションを読まずに（`_protected` 外で）描画される。

「ログイン済み来訪者がアプリから放り出された感」を解消したくなるが、ここでシェルを被せる＝閲覧者によって出し分けると、このページの「公開・中立・キャッシュ可能」という性質を自ら壊す。一方で「戻れない」のは純粋な実装上の欠陥（後述）。

## 決定

### 1. 公開ページは閲覧者非依存。ログイン状態で見せ方を変えない

`/u/{urlId}/p/{token}` は**ログインの有無・閲覧者が誰かに関わらず、常にクローム無しの同一表示**とする。自動ログインしていても見え方は変えない。アプリシェル（ヘッダー/ボトムナビ）は被せない（ADR-0018 §6 を踏襲）。

理由：

- OGP/SEO/キャッシュの一貫性 — 閲覧者ごとに出し分けると、クロール・シェアされる正本ビューが揺れる。
- 中立性 — 「自分が今ログイン中の誰々として見ている」という文脈を、本来中立であるべき公開名刺ビューに持ち込まない。
- 単純性 — このルートに認証セッション読み取りを持ち込まない（`_protected` 外のまま）。

### 2. 「アプリから放り出された感」は戻る導線で解く

シェルを被せる代わりに、**「← 戻る」を確実に出す**ことで足りるとする。表示判定を `document.referrer` の同一オリジン判定から TanStack Router の `useCanGoBack()`（履歴ベース）へ置き換える。SPA 内部遷移（公開→公開の「他のなふだ」、イベントページ→プロフィール等）で履歴があれば必ず戻れ、QR からの新規コールドロードでは履歴が無いので出さない（=外部から来た人には戻り先が無い、という想定どおり）。

### 3. オーナーの `/me` 内「他のなふだ」は in-place 切り替えにする

[[なふだリンク]]は定義上**必ず自分の別ペルソナ**を指す（ADR-0015）。よって `/me` に並ぶ「他のなふだ」チップの遷移先は、必ず上部 PersonaSwitcher に存在するペルソナである。`/me` ではチップを公開ページ遷移ではなく **PersonaSwitcher と同じ in-place 切り替え**（`targetPersonaId` で切替）にする。これによりオーナーはシェルから出ず、「ヘッダー/フッター/戻りが無い」問題が発生源ごと消える。

`/me` の切り替えビューは「オーナー視点」（非公開フィールドが薄く＋🔒で見える管理ビュー）であり、公開ページの「真の来訪者ビュー」（非公開は完全非表示）とは別物。`/me` で見たいのは通常この管理ビューなので、これで正しい。「来訪者に実際どう見えるか」の確認は「なふだを見せる」（QR/閲覧URL）が担う。

## 検討した代替案

- **ログイン済み来訪者にはアプリシェルを被せる**: 放り出され感は消えるが、同一ルートを外部スキャナーも開くため、別ルート化か「オーナー/ログイン判定して出し分け」が必要になり、§1 の公開・中立・キャッシュ性を犠牲にする。複雑度が見合わない。
- **`/me` のチップも公開ページへ飛ばし、戻るだけ直す**: オーナーは「真の公開ビュー」をプレビューできるが、毎回シェル外のクローム無しページに落ちる体験は `/me` が「自分の管理プレビュー」である前提と噛み合わない。in-place 切り替えの方がオーナーの意図に近い。
- **来訪者がオーナー本人なら編集導線を出す**: 将来のオプションとしては有り得るが、公開ページに認証判定を持ち込む点で §1 と衝突する別機能。今回のスコープ外。

## 結果・トレードオフ

- 公開ページはログイン状態に依存しない単一の閲覧専用ビューとして固定され、OGP/SEO/キャッシュの一貫性を保つ。
- ログイン済み来訪者は公開ページでアプリシェルを得られないが、`useCanGoBack()` による「← 戻る」でアプリ内遷移からは必ず戻れる。コールドロード時は戻り先が無いのが正しい挙動。
- `/me` の「他のなふだ」はシェル内 in-place 切り替えになり、オーナーは公開ページへ落ちなくなる。`NafudaLinkChip` は公開ページではリンク、`/me` ではコールバック、の2モードを持つ。
- なお `EmptyState` 等も同じ `document.referrer` 判定を持つが、本 ADR の対象は公開プロフィールに限定する（横展開は別途）。
- 「公開プロフィール」「アプリシェル」は UI/ルーティング語彙でありドメイン概念ではないため `CONTEXT.md` には登録しない。
