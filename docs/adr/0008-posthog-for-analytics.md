# ADR-0008: アナリティクスに PostHog を採用する

- **ステータス**: 採用
- **決定日**: 2026-06-05

## 背景

サービスのコアバリューである「QRスキャン→コネクション成立」が実際にどれほど機能しているかを計測する手段がなかった。改善の根拠を作るために、以下の2ファネルを計測できるアナリティクスを導入することにした。

- **登録ファネル**: `トップ到達 → ログイン → ウィザード完了`
- **QR交換ファネル**: `つながりQR表示 → スキャン（connect ページ到達） → 「つながる」タップ → コネクション成立`

## 決定

アナリティクスツールとして **PostHog（US Cloud）** を採用する。

- `posthog-js` を `src/lib/analytics.ts` にラップし、本番環境（`import.meta.env.PROD`）のみ初期化する
- `persistence: 'localStorage+cookie'`、`ip: false`（IP匿名化）で設定し、Cookie同意バナーを不要にする
- プライバシーポリシーページを同時に作成し、解析ツールの使用を明示する

## 却下した代替案

- **Google Analytics 4**: ページビューは得意だが、カスタムファネルの設定が煩雑（「探索」レポートの習熟コストが高い）。また Cookie 同意バナーが必要になる可能性がある
- **Cloudflare Web Analytics**: すでに Cloudflare 上にいるためゼロコストで導入できるが、カスタムイベントによるファネル分析が不可能

## トレードオフ

- **失うもの**: GA4 の知名度・エコシステム（日本語の事例が豊富）。後から GA4 に移行すると歴史的データが失われる
- **得るもの**: カスタムファネルの直感的な設定。月100万イベントまでの無料枠でしばらく間に合う。Cookie バナーが不要なシンプルなプライバシー対応
- **許容できる理由**: このサービスの改善サイクルに必要なのは「どのステップで離脱しているか」という行動ファネルであり、PostHog の方がそのユースケースに直接対応している
