# ADR-0014: ユーザー画像は背景画像カスタマイズではなく独立した「ギャラリー」コンテンツとして追加する

- **ステータス**: 採用
- **決定日**: 2026-06-15
- **関連**: ADR-0006（なふだスタイルはIDのみDB保存・定義はコード管理）の思想を維持する

## 背景

「Linktree は写真を出せないのが残念」という実ユーザー（若者）の声を起点に、なふだにアバター（プロフィール画像）以外の写真を載せられるようにしたいという要望が出た。当初は2つの方向で語られた：

- **(A) 背景画像を設定可能にする** — なふだスタイルの一環として、ユーザーが背景画像をアップロードできるようにする。
- **(B) プロフィール画像以外に写真を表示できるようにする** — 推し・作品・愛車などの「対象物」を写真で見せる。

ヒアリングの結果、本当の動機は **(B) 対象物を画で見せて出会いのきっかけにする**ことであり、「自分らしい背景にしたい」という自己表現欲求ではないと確定した。

## 問題

(A) の「背景画像をユーザーがアップロードして設定する」は、**なふだスタイルの設計思想と正面衝突する**。

なふだスタイルは ADR-0006 で「**IDのみDB保存・定義はコード管理・パッケージまるごと選択（個別カスタマイズ不可）**」と決定済みで、却下した代替案がまさに「ユーザーごとに値を持つカスタマイズ方式」だった。背景画像のアップロードは「個別要素のカスタマイズ」そのものであり、これを認めると：

- スタイル定義の一元管理・全ユーザーへの一括反映・有料フラグのコード集約という ADR-0006 の便益が崩れる。
- 「装飾（コード管理）」と「コンテンツ（ユーザー入力）」のレイヤーが混ざる。

一方で (B) の動機は装飾ではなく**コンテンツ**（アバターと同系列の、ユーザーが載せる情報）であり、なふだスタイルに手を入れずに実現できる。

## 決定

**(A) 背景画像のカスタマイズは採らない。(B) を、なふだスタイルとは別レイヤーの独立コンテンツ「ギャラリー」として追加する。**

- ギャラリーは何にも紐づかない独立した写真の1セット（推しタグ・SNSリンクとは無関係）。`gallery_photos` テーブルに格納し、`sns_links` と同じパターン（persona に複数紐づき・表示順あり）で実装する。
- 各写真は任意の短いキャプション（空可）を持てる。**最大6枚**。
- 公開プロフィールでは正方形サムネイルのグリッド＋タップで全体表示（ライトボックス）。配置は推しタグの直後・SNSリンクの前。
- [[公開範囲]]（`field_visibility`）の対象とし、**ギャラリー全体で1トグル**（`gallery` キー、デフォルト公開）。
- 保存は既存のアバター経路（R2 バケット・JPEG・data URL アップロード）を流用し、容量上限のみ引き上げる（長辺〜1080px・1枚〜600KB）。
- なふだスタイルは引き続き「コード管理の装飾」に限定し、ユーザーがアップロードする画像は持たない。

## 却下した代替案

- **(A) 背景画像のカスタマイズ**: ADR-0006 が却下した「ユーザーごとに値を持つ」方式を裏口から呼び戻す。装飾の一元管理・一括反映・有料ゲートという便益を失い、装飾とコンテンツのレイヤーが混ざる。動機（対象物を見せる）にも過剰で、ずれている。
- **写真を推しタグ／SNSリンクに紐づける**: タグは文字（アーティスト名等）の集合で写真と 1:1 にならず、両方の入力が重くなる。独立テーブルにしておけば将来の紐づけ拡張も妨げない。
- **写真ごとの公開/非公開トグル**: 最大6枚に対して過剰。「見せたくない時期は丸ごと隠す」で十分。

## トレードオフ

- **失うもの**: 背景を自分の画像で飾る自己表現の自由度。なふだスタイルは引き続きコード提供のパッケージから選ぶ方式に留まる。
- **得るもの**: ADR-0006 の「装飾はコード管理のパッケージ」という背骨を守ったまま、実ユーザーが求めた「対象物を画で見せる」を満たせる。実装は実績ある `sns_links`／アバター経路の流用で一貫し、コストは「6枚 × なふだ数」でハードに頭打ちになる。
- **許容できる理由**: ヒアリングで動機は自己表現（背景）ではなく対象物の提示（コンテンツ）だと確定した。装飾のカスタマイズ自由度は打ち出し（ADR-0009：その場で会った人とSNSでつながる）のコア価値ではなく、ここで ADR-0006 を曲げる見返りに乏しい。コンテンツとして足す方が動機にも既存設計にも整合する。
