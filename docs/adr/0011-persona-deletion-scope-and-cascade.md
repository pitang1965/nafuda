# ADR-0011: なふだ削除のスコープとカスケード

- **ステータス**: 採用
- **決定日**: 2026-06-13

## 背景

なふだ（ペルソナ）は追加・編集できるが、**1枚だけ削除する**手段がなかった。ユーザーが不要になったなふだ（例：使わなくなった「婚活用」）を整理できるよう、編集画面に確認付きの削除機能を設ける。

これは既存の **退会 (Account Deletion)**（ADR-0005）とは別概念である。退会はアカウント・全なふだ・UrlId を完全削除するが、なふだ削除は1枚だけを消し、アカウント・UrlId・他のなふだは残す。

## 問題

なふだ削除の設計にあたり、3つの分岐を解決する必要があった。

1. **最後の1枚を削除できるか**。なふだを0枚にすると「アカウント有り・UrlId 有り・なふだ0枚」という中途半端な状態が生まれる。`/me` はこの状態でウィザードへリダイレクトするが、UrlId は残り、コネクション相手の行は宙に浮く。

2. **コネクションのカスケードが「非対称削除」原則と衝突する**。CONTEXT.md のコネクション定義には「削除は非対称で、各自が自分の行を独立して削除できる（相手の行には影響しない）」とある。一方 `connections` の FK は `fromPersonaId`・`toPersonaId` とも `personas.id` を onDelete 指定なしで参照する。A↔B のコネクションは Row1(`from=A,to=B`) と Row2(`from=B,to=A`) の2行から成り、なふだ A を削除すると Row2 が `to=A` で削除対象を参照するため、**Row1 だけ消して Row2 を残すことは DB レベルで不可能**。

3. **デフォルトなふだを削除したときの不変条件**。CONTEXT.md は「1ユーザーにつき1つが `isDefault = true`」を宣言している。デフォルトを削除すると不変条件が崩れる。

## 決定

### 最後の1枚は削除させない

なふだ削除は**2枚以上あるうちの1枚を消す操作に限定**する。最後の1枚は削除不可とし、UI ではボタンを非表示にして「アカウントごと消す場合は退会してください」と案内する。サーバー側でも最後の1枚の削除を拒否する（多層防御）。全削除の意図は退会が担う。

### コネクションは相手側の記録も含めて両行削除

なふだ削除時、当該なふだが `from` または `to` に含まれるコネクションを**両行とも削除**する（ADR-0005 の退会カスケードと同じ挙動）。相手への通知なし・ゴースト表示なし（ADR-0005 でゴースト表示は却下済み）。

CONTEXT.md の「非対称削除」原則は **コネクション1件を解除する（つながりを切る）操作** の定義であって、**なふだという主体そのものを消す操作** には及ばない。なふだが消滅すれば、それに紐づくコネクションは相手の鏡像行も含めて消えるのが自然であり、退会で既に受け入れている挙動である。

### カスケード（削除するもの）

| エンティティ | 処理 |
|---|---|
| `connections` | `fromPersonaId` または `toPersonaId` が当該なふだを指す行をすべて削除（相手側の記録も含む） |
| `event_checkins` | 当該なふだのチェックイン記録を削除 |
| `sns_links` | 当該なふだに紐づく SNS リンクを削除 |
| `connection_qr_tokens` | FK の `onDelete: cascade` で自動削除 |
| `personas` | 当該なふだを削除 |

### 保持するもの（削除しないもの）

| エンティティ | 処理 | 理由 |
|---|---|---|
| `events`（企画・即時とも） | 削除しない。`hostUserId` そのまま、`hostPersonaId` も放置 | 退会と違い本人は残るため所有(`hostUserId`)は有効。`hostPersonaId` は表示・アクセス制御のどこからも読まれず無害 |
| `url_ids`・`user`・他のなふだ | 保持 | アカウントは存続する |

### デフォルトなふだの再選定

削除対象が `isDefault = true` だった場合、残るなふだのうち**最古（`createdAt` 順の先頭）に `isDefault = true` を付け替える**。これにより「常に1つがデフォルト」という不変条件を維持する。削除対象がデフォルトでなければデフォルトは触らない。

## 却下した代替案

- **最後の1枚も削除できてウィザード未設定状態に戻す**: 「なふだ0枚・アカウント有り・UrlId 有り」という中途半端な状態を新たに作り、全消し責任が退会フローとなふだ削除フローに分散する。誤読・実装の重複を招くため却下。全消しは退会に一本化する。
- **コネクションを自分の行(Row1)だけ削除して相手の行(Row2)を残す**: FK 制約上不可能。仮に可能でも、消えたなふだを参照する宙ぶらりんの行が相手の一覧に残り UX として不自然。
- **`hostPersonaId` を NULL 更新する**: どこからも読まれない列であり、退会後も同列は宙に浮くが問題化していない。あえて更新処理を足さない（最小変更）。

## トレードオフ

- **失うもの**: なふだ削除の取消手段（即時削除のため復元不可）。相手側のつながり記録も消えるため、相手から見ると「つながっていた相手が一覧から消える」。
- **得るもの**: なふだ整理のシンプルな手段。退会との責任分界が明確（1枚＝なふだ削除／全部＝退会）。
- **許容できる理由**: 確認モーダル（消えるもののリスト提示）で誤操作を防ぐ。退会ほど壊滅的でない（アカウント・UrlId・他のなふだは残り、なふだは作り直せる）ため、退会のような同意チェックボックスまでは課さない。
