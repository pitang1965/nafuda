# ADR-0015: なふだリンクは所有ペルソナへの内部ID参照とし、SNSリンクからはなふだURLを排除する

- **ステータス**: 採用
- **決定日**: 2026-06-16

## 背景

オーナーが自分のなふだに、自分が持つ別のなふだ（例：「婚活用」→「趣味用」）への導線を貼れるようにしたい、という要望が出た。依頼当初は「SNSリンクの一種として貼る」か「独立した『なふだリンク』として持つ」かが未確定だった。

`CONTEXT.md` のグロッサリーは [[SNSリンク]] を「ペルソナに紐づく**外部サービス**へのリンク」と定義している。一方で今回貼りたいのはアプリ内部の別なふだへの参照であり、外部サービスではない。

## 問題

なふだ→なふだの内部リンクをどう表現するか、大きく2つの軸で決める必要があった。

1. **概念**: 既存の `sns_links` に新プラットフォーム（例 `nafuda`）を足すか、独立した `nafuda_links` 概念にするか
2. **参照方法**: リンク先を `/u/{urlId}/p/{token}` の **URL文字列**で持つか、**`targetPersonaId`** で構造化参照するか

加えて、なふだのアクセス制御は [[ShareToken]] が単独で担う（推測耐性）。SNSリンクは任意URLを受け付けるため、**他人のなふだのプロフィールURLを本人の同意なく自分のSNSリンクに貼ると、その他人の ShareToken を公開してしまう**という露出経路が存在する。

## 決定

**なふだリンクを独立概念とし、リンク先を `targetPersonaId` で内部参照する。あわせて SNSリンクのURLにはなふだのプロフィールURLを登録できないようにする。**

- `nafuda_links` テーブルを新設し、`personaId`（リンク元）・`targetPersonaId`（リンク先）・`displayOrder` を持つ
- リンク先はオーナー自身の `userId` のペルソナのみをピッカーから選ぶ（**他人のなふだは指せない**＝所有権が構造的に保証される）
- 公開プロフィールでは、リンク先ペルソナから **displayName とアバターを動的取得**してコンパクトなチップ（アバター丸＋名前、アバター無しは 📛 にフォールバック）で横並び表示する
- `personaId`・`targetPersonaId` 双方に `onDelete: cascade` を張り、リンク元・リンク先どちらが消えても幽霊リンクが残らないようにする
- SNSリンクのURL検証で、なふだのプロフィールURL形式（`/u/{urlId}/p/{token}`）を**自分・他人を問わず拒否**する。内部参照はなふだリンクに一本化する

なお、自分の別のなふだを公開プロフィール経由で繋ぐことによる ShareToken 露出（例：婚活用→推し活用を見せた相手全員が推し活用も開ける）は、警告も公開範囲トグルも設けない。プロフィールQRは元々「インターネット公開・印刷も可能」な設計であり、自分の別のなふだを繋ぐのは**オーナーによる自己公開の一種**にすぎないため。

## 却下した代替案

- **SNSリンクの新プラットフォームとして実装**: 既存テーブル・UIを再利用できるが、グロッサリーの「SNSリンク＝外部サービス」定義を濁す。また所有権検証・ライブ表示・被参照カスケードといった内部リンク固有の要件を `sns_links` に後付けすることになり、概念がにじむ。
- **URL文字列で参照**: 実装は軽いが、「📛なふだ名」を出すには名前を別途手入力する必要があり、リンク先の表示名変更が反映されず、リンク先削除後に404の幽霊リンクが残る。所有権検証もURLパースに頼ることになる。
- **他人のなふだも貼れるようにする**: 本人の同意なく他人の ShareToken を公開する経路になるため、要望（「自分の他のなふだ」）の範囲を超えて拒否した。

## トレードオフ

- **失うもの**: 他人のなふだを貼る自由、SNSリンクに任意URLとしてなふだを貼る柔軟さ、URL方式の実装の軽さ
- **得るもの**: 所有権の構造的保証、リンク先の名前・アバターの常時最新表示、削除時の整合性（幽霊リンクが出ない）、他人の ShareToken 露出の防止、グロッサリーの「外部／内部」境界の明確さ
- **許容できる理由**: 要望は「自分の他のなふだへのリンク」であり、他人参照や任意URLは射程外。内部参照に絞ることで得られる堅牢さのほうが、捨てる柔軟さより価値が高い。
