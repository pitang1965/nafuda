# ADR-0010: 用途タイプ(Purpose)をラベルと分離した機械可読スイッチとして導入する

- **ステータス**: 採用
- **決定日**: 2026-06-13

## 背景

なふだ（ペルソナ）には用途ごとに見せ方を変えたいという要求がある。推し活なら推しタグを「推し」として前面に出し、車趣味ならみんカラ（minkara）を SNS リンクのサジェスト筆頭にしたい、といった具合に、用途に応じて公開プロフィールの表示や編集画面の体験を切り替えたい。

既存の `personas.label` は「なふだを自分だけが識別するための自己識別名」（「推し活用」「婚活用」など自由記述・非公開）であり、用途の概念を一見カバーしているように見える。しかし label は**人間が読む任意の文字列**であって、アプリが分岐の判断に使える値ではない（表記揺れ・自由記述ゆえに機械可読でない）。

そこで「アプリが見せ方を切り替えるために読む、機械可読な用途の分類」を新しく持つかどうかを検討した。

## 問題

「用途に応じた見せ方の切り替え」をどのデータ構造で実現するか。主な論点は3つ:

1. 既存の `label` を流用するか、新しい概念（purpose）を分離して持つか
2. 新設する場合、カラム型を PostgreSQL ネイティブ enum にするか `text` + アプリ側レジストリにするか
3. 既存なふだの後方互換をどう担保するか（全件バックフィルするか、null を許すか）

## 決定

**`personas` に `purpose`（`text`・null許容）を新設し、ラベルとは別の「機械可読な用途分類」として扱う。許容値はアプリ側の TypeScript レジストリで管理し、null を「導入前の既存なふだ」専用の状態とする。**

- **label とは分離する**: label は"オーナーが呼ぶ名前"（自己識別・非公開・自由記述）、purpose は"アプリが読む種別"（見せ方・編集体験の分岐に使う有限分類）。役割が異なるため別カラムにする。purpose の効果はオーナー向け画面だけでなく公開プロフィール（タグ見出し・SNSサジェスト順）にも波及しうる。
- **型は `text` + アプリ側レジストリ**: 特化した用途（例：`car`、将来の他ジャンル）を継続的に追加していく前提のため、追加のたびに `ALTER TYPE` を要する PG ネイティブ enum を採らない。既存コードベースの慣行（`snsLinks.platform`、`personas.styleId` がいずれも `text` + コード側管理）とも揃える。見せ方の分岐は「purpose → 表示設定」のレジストリ参照で実装し、未知値・null は default にフォールバックする。
- **初期値セット（開始時の値）**: `oshi`（推し活）/ `social`（イベント・交流）/ `car`（車）/ `other`（その他）。`car` は「みんカラをサジェスト筆頭にできる唯一の用途」かつ既存グロッサリの実例（「車趣味用なふだ」）であり、固有の挙動差を持つため最初から含める。
- **null = 既存なふだ専用**: 全件バックフィルはせず、導入前のなふだは null のまま＝従来表示（1px も変えない）。ウィザードは用途選択を必須にする（最低でも `other`）ため、新規作成されるなふだは常に non-null。これにより「null＝導入前の既存」という不変条件が保てる。`other` は表示上 null と同等だが「明示的に選択済み」という点で意味が異なる。
- **編集での変更を許可**: purpose はプロフィール編集画面で変更でき、既存の null なふだもそこからオプトインで用途を選べる。purpose を変えても保存済み label は書き換えない（label は未編集のときだけ purpose 由来の初期値を seed する）。

## 却下した代替案

- **label を流用する（purpose を新設しない）**: label は自由記述で表記揺れが避けられず、アプリが分岐に使えない。「推し活用」「推し活」「推し」を同一視するロジックは脆く、ユーザーの自己識別の自由を奪う。役割の異なる2つの関心（人間が呼ぶ名前／アプリが読む種別）を1カラムに混ぜることになり、CONTEXT.md 上の語衝突（どちらも「用途」）も解消できない。
- **PostgreSQL ネイティブ enum**: 型レベルの保証は得られるが、特化用途を足すたびにマイグレーション（`ALTER TYPE ... ADD VALUE`）が必要で、並べ替え・削除も事実上できない。「用途を継続的に増やす」という前提と相性が悪く、既存コードベースの text 慣行とも食い違う。
- **導入時に全件バックフィル**: 既存なふだに既定 purpose を一括付与すると、「ユーザーがまだ用途を選んでいない」状態を表現できなくなり、意図しない見せ方の変化（見出し追加など）を既存ユーザーに押し付けるリスクがある。null を「未選択／導入前」として残す方が、互換性とオプトイン移行の両方を満たせる。

## トレードオフ

- **失うもの**: `text` 採用によりDBレベルの値保証はなくなる（アプリ側バリデーションに依存）。label と purpose という似て非なる2フィールドが並ぶため、将来の読者が混同する余地が残る（CONTEXT.md の区別表で緩和）。
- **得るもの**: 用途の追加がマイグレーション不要の設定変更に落ちる。見せ方の分岐がレジストリ参照に集約され `if/else` の散乱を防げる。null=既存の不変条件により後方互換が保証され、既存ユーザーへの非破壊なオプトライン移行ができる。label がユーザー所有の自由記述として純化される。
- **許容できる理由**: 値の追加頻度とコードベースの一貫性を考えると、型保証より拡張容易性の価値が高い。2フィールドの混同リスクは文書とUI上の役割分担（ラベル＝自己識別／用途タイプ＝アプリのスイッチ）で十分に抑えられる。
