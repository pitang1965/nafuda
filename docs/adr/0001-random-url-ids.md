# ADR-0001: UrlId をシステム生成のランダム文字列にする

- **ステータス**: 採用
- **決定日**: 2026-05-27

## 背景

ユーザーのプロフィールURL `/u/{urlId}` の `urlId` 部分を、ユーザーが自由に設定できる文字列（例: `pitang1965`）にする設計が初期に存在した。

## 問題

ユーザーが選べる UrlId はプライバシーリスクをはらむ。

- ターゲットユーザー（推し活ユーザー）はハンドルネーム運用を前提としており、プライバシーへの意識が高い
- 自分の X ハンドルと同じ文字列を UrlId に設定してしまうと、第三者が `nafuda.app/u/{Xハンドル}` を推測してデフォルトなふだを閲覧できる
- UrlId は変更不可のため、一度公開されると取り消せない

## 決定

UrlId はアカウント作成時にシステムが生成するランダム英数字文字列とする。ユーザーは設定・変更できない。

URL 構造は変更しない：
- デフォルトなふだ: `/u/{randomUrlId}`
- 特定のなふだ: `/u/{randomUrlId}/p/{shareToken}`

なふだの識別名（「自動車趣味用」「推し活用」等）は displayName フィールドで管理し、URL には反映しない。

## 却下した代替案

- **ユーザー選択式 UrlId**: 推測可能・列挙可能なため却下
- `/u/{urlId}/{ラベル}` 形式の追加なふだ URL: ラベルが推測可能なため却下。追加なふだの識別は既存の shareToken（32桁ランダム16進数）で十分

## トレードオフ

- **失うもの**: URLを口頭で伝える手段（`nafuda.app/u/pitang1965` のような覚えやすさ）
- **得るもの**: UrlId からユーザーのSNSアカウント等が推測されないプライバシー保護
- **許容できる理由**: 推し活の主要シナリオはQRコードでの交換であり、URLを口頭で伝える必要性は低い
