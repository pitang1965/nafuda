# ADR-0022: つながり検知・ライブ presence を Durable Objects の realtime 基盤で実現する（QR の 3 秒ポーリングを廃止）

- **ステータス**: 採用
- **決定日**: 2026-06-27
- **関連**: ADR-0017（staging/dev DB 環境分離）の環境結線方針を踏襲（preview/staging/本番ごとに共有シークレットを別管理）。ADR-0012（コネクション文脈の独立コピー）が定める「Postgres が正本」の原則を realtime 層にも適用。

## 背景

「なふだを交換する」で QR を見せた側（A）が接続成立を検知する仕組みが、`me.tsx` の 3 秒間隔ポーリング（`setInterval(poll, 3000)` → `checkQrConnectionStatus`）になっていた。WebSocket も SSE も使わず、QR 表示中ずっとサーバーを叩き続ける。

きっかけは「3 秒ポーリングは筋が悪い」という**設計上の引っかかり**であって、計測された UX 障害ではない。検討の結果、争点と射程を次のように整理した：

- **バックエンド負荷は実問題ではない**。ポーリングはシートを開いている本人・対面交換中の数秒〜数十秒だけ動き、大半は「トークン存在 → 即 `pending`」の軽いクエリ。会場規模でも負荷は些少。むしろ輻輳した会場電波下では常時接続より頑健ですらある。
- **最大 3 秒の体感遅延**も、対面では実害が小さい（成立は B のタップで確定し、A のポーリングは確認トーストとシートクローズのためだけ）。

したがって本決定は「QR の遅延を直す」ためではなく、**なふだに realtime 基盤を導入する**ための投資として位置づける。本命のユースケースは QR 確認ではなく**イベントのライブ presence（いま会場にいる人・出入りのリアルタイム反映）**で、QR 確認はその基盤に薄く相乗りさせる。

## 問題

realtime 基盤を Cloudflare 上に載せるにあたり、次の硬い制約と設計分岐があった：

1. **Durable Object クラスは Pages プロジェクトから定義（export）できない**。現行 `nafuda` は Pages デプロイ（`pages_build_output_dir`）。DO を入れる瞬間、ホスティング形態の分岐が生じる。
2. presence（多人数・相互可視）と QR 確認（1:1・点対点・本人宛通知）は**形が違う**。同じ部屋に押し込めるか、別系統にするか。
3. コンパニオン Worker は本体と**別トラスト境界**。better-auth のクッキーセッションを別 Worker でどう扱うか。
4. push は撃ちっぱなしになりうる。会場の輻輳で受信側 WS が切れた瞬間の取りこぼしをどう収束させるか。
5. 「3 秒ポーリングが筋悪い」を消したいが、WS が張れない会場で何を保険にするか。

## 決定

**1. DO はコンパニオン Worker 先行。Pages → Workers 全面移行とは切り離す。**

- DO クラスを独立した小さな Worker としてデプロイし、現行 Pages からバインドして利用する。本体アプリ（TanStack Start on Pages）は無傷のまま realtime 部分だけを独立して育てられる（爆発半径の最小化）。
- Pages → Workers (static assets) 全面移行は「realtime が基盤として定着した」と確信できてからの別建ての中長期タスクとし、必要なら独立 ADR で判断する。**コンパニオン先行は捨て作業にならない**——移行時は DO クラスを本体 Worker 内へ移し、バインディングを cross-script から same-script に変え、クライアントの WS URL を差し替えるだけ。

**2. DO クラスは 2 種類。`EventRoom`（presence）と `PersonaChannel`（push 通知）。**

- `EventRoom` = `idFromName(eventId)`。1 イベント = 1 部屋。check-in / out を部屋の全員へブロードキャストし、ライブ参加者一覧を差分更新する。
- `PersonaChannel` = `idFromName(personaId)`。「このなふだに起きた出来事」を本人へ push する恒久チャンネル。**QR 確認はここに相乗りさせる**（per-token の使い捨て DO は作らない）。将来は保留招待の後追い成立（48h 後に非同期で成立＝ポーリングでは拾えない realtime の本命）など、コネクション系通知の受け皿として射程が広がる。

**3. クライアント↔DO の通信は WebSocket（DO Hibernation 前提）一本。**

- presence の双方向性に最も素直。Hibernation でアイドル時のコストを圧縮。SSE との二方式混在はしない。

**4. 認証はチケット方式、サーバー→DO push は Service Binding。DO は DB/セッション検証を持たない薄い中継に徹する。**

- 本体の server fn `issueRealtimeTicket({ personaId, eventId? })` を新設。セッションを持つ本体が本人確認し、`personaId` / スコープ / 短い `exp`（60 秒程度）を含む HMAC 署名トークンを発行する。
- クライアントは `wss://<companion>/?ticket=...` で接続。コンパニオン Worker は**共有シークレット（両デプロイの env var）で署名検証するだけ**で DB に触れず本人性を確定し、対応する DO へルーティングする。
- `createConnectionFromQr` などが Postgres に成立を書いた**後**、本体から **Service Binding**（オリジン外に出ない内部 RPC、内部シークレットヘッダ付き）でコンパニオン Worker を叩き、`PersonaChannel(issuerPersonaId)` へ push させる。
- 認証の正本は常に本体（better-auth ＋ DB）に残る。

**5. Postgres を正本に固定。DO はステートレス中継。クライアントは (再)接続のたびに正本を 1 回読んで収束する（reconcile-on-connect）。**

- WS が開いた瞬間に `checkQrConnectionStatus`（since 付き）を 1 回叩き、接続前に起きた成立を回収。接続中は push に乗り、切れて再接続したらまた since で読む。取りこぼしは再接続ごとに自己修復する。
- `EventRoom` も同型：DO は名簿を保持せず check-in/out の中継だけ。名簿の正本は `event_checkins`。クライアントは (再)接続時に名簿を再取得し、以後 push で差分更新する。これにより DO のコールドスタート（hibernation 復帰・退避）で復元処理が不要になる。

**6. 平常時は WS のみ。3 秒ポーリングは廃止。WS が張れない時だけ縮退ポーリングに落とす。**

- WS が N 回張れない場合のみ、同じ reconcile 用エンドポイントを 10〜15 秒間隔で叩く graceful degradation に落とす。3 秒ポーリングの復活ではなく、WS 不成立時限定・低頻度・既存エンドポイント再利用の保険。新エンドポイントは増やさない。

## 却下した代替案

- **ポーリング据え置き（何もしない）**: 負荷も遅延も実害は小さく、純粋な技術的費用対効果ではこれが妥当。だが realtime 基盤（本命=ライブ presence、保留招待の後追い成立）を持たない限り、ポーリングでは綺麗に書けない機能が残り続ける。基盤投資としての価値を採って退けた。
- **QR 確認だけを SSE で realtime 化**: 最小前進だが、presence の上り方向（自分の入室通知）を別経路にする非対称構成になり、QR 専用の使い捨て機構が残って汎用化しない。基盤を作る方針（決定 2）と相容れない。
- **per-token DO（`idFromName(qrToken)`）**: QR 表示ごとに短命 DO を作る案。隔離は綺麗だが、保留招待の後追いやイベント外通知を拾えず、QR 専用機構が残る。`PersonaChannel` への相乗りなら同じ push を汎用チャンネルの最初の一利用に格上げできる。
- **粒度を user（`idFromName(userId)`）に集約**: ルーティングは楽だが、サーバーが知るのは `issuerPersonaId` であり persona 粒度の方が素直で、複数なふだ運用とも噛み合う。
- **Pages → Workers 全面移行を先にやる**: realtime の前に無関係なデプロイ移行リスクを背負うことになる。コンパニオン Worker 先行で本体を無傷に保ち、移行は分離した（決定 1）。
- **DO にセッション検証 / DB アクセスを持たせる**: コンパニオン Worker に auth 秘密や DB を持ち込むと信頼境界が二重化し、DO が太る。チケット署名検証だけに絞り、正本を本体に残す方が薄く安全（決定 4）。
- **DO を正本にする / DO に再送キューを持たせる**: push の取りこぼしを DO 側のキューで解こうとすると DO が状態を持ち太る。Postgres 正本＋ reconcile-on-connect なら DO を薄いまま自己修復できる（決定 5）。

## トレードオフ

- **得るもの**: ライブ presence という本質的に realtime なユースケースに素直な基盤を得る。QR 確認は「専用ポーリング」から「汎用通知チャンネルの一利用」へ格上げされ、当初の「筋が悪い」が解消する。保留招待の後追い成立など、ポーリングでは原理的に拾えない通知の受け皿ができる。会場の輻輳には reconcile-on-connect ＋縮退ポーリングで頑健に degrade する。
- **失うもの／代償**: (a) デプロイ単位が 2 つに増える（本体 Pages ＋ コンパニオン Worker）。preview/staging/本番ごとの Service Binding 結線と共有シークレット管理が増える（[[ADR-0017]] と同じ環境分離の注意が必要）。(b) 新しい障害モード（WS 切断・チケット失効・DO コールドスタート）と、それを吸収する reconcile 経路の実装・テストが要る。(c) Cloudflare の DO 課金が新たに発生する（Hibernation で圧縮はするが）。
- **許容できる理由**: 爆発半径はコンパニオン Worker 先行で最小化し、本体は無傷。DO を「正本にしない薄い中継」に徹させることで、増える複雑さの大半（一貫性・復元）を Postgres 正本＋ reconcile-on-connect という 1 つの原則に畳み込める。費用対効果は QR 単独では合わないが、ライブ presence を本命に据えた基盤投資としては正当。
