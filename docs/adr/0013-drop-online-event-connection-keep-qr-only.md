# ADR-0013: オンライン接続（イベント参加者一覧からのつながりリクエスト）を設計から削除し、QR一本化を維持する

- **ステータス**: 採用
- **決定日**: 2026-06-14
- **関連**: ADR-0007 の経路③（イベントリクエスト）を撤回する（ADR-0007 を一部改訂）

## 背景

ADR-0007 は「つながる」を3経路（永続URL＝接続なし／つながりQR＝対面交換／イベントリクエスト＝参加者一覧からの申請・承認）に整理した。このうち実装されたのは**つながりQR経由（`createConnectionFromQr`）のみ**で、イベントリクエスト（CONTEXT.md 上の「つながりリクエスト」）と保留中の招待は未実装のまま残っていた。

「その場で会ったのにQRを交換しそびれた相手と、後からイベント参加者一覧でつながりたい」という取りこぼし救済（以下シナリオA）を実現すべきか——すなわち**オンラインだけで成立する接続が本当に必要か**を問い直した。

検討の結果、以下が判明した。

- イベントへのチェックイン（`checkinToEvent`）は **ログイン済み・ペルソナ所有・イベントの `shareToken` を知っていること**しか要求せず、**GPS近接や時間窓による実在検証がない**（`gpsCoordinates: null`）。
- イベントの公開ページ（`/e/{slug}`）は誰でも開け、ワンタップの「参加する」で参加者一覧に載れる。
- したがって参加者一覧は「**会場に物理的にいた人**」ではなく「**この公開URLを開いて参加するを押した人**」の集合にすぎない。

## 問題

シナリオAを成立させるには、参加者一覧の上に「つながりリクエスト（申請・承認）」を載せる必要がある。しかし参加者一覧が実在証明になっていないため、これは**会っていない相手からの申請**を構造的に許す。これは ADR-0007 が公開URL接続を却下した理由——「見知らぬ人が勝手にコネクション一覧に入るスパム構造」——を、A（取りこぼし救済）の皮をかぶった B（遠隔発見）として裏口から呼び戻すことになる。

選択肢は本質的に3つ：

1. **チェックインに実在ゲート（GPS近接＋時間窓）を追加する** — 参加者一覧を本物の在席証明に格上げし、つながりリクエストを安全に成立させる。実装コスト大・GPS許可必須・屋内精度問題。
2. **ゲートせず、承認制＋失効＋同担拒否で緩和する**（ADR-0007 の元案） — コネクションは勝手に増えないが、未承諾の「申請スパム」は残る。
3. **オンライン接続を作らない** — つながりQRの対面交換に一本化する。取りこぼしは「その場でQRを出し合う」運用と、参加者一覧から相手の**永続プロフィールを閲覧できる**現状（接続はしない）で受け止める。

## 決定

**方針3を採用する。オンライン接続（イベント参加者一覧からのつながりリクエスト）を設計から削除し、コネクション生成をつながりQRの対面交換に一本化する。**

- CONTEXT.md から「つながりリクエスト」の用語を撤去し、コネクションの生成経路を「なふだを交換する（つながりQR経由）」のみと定義する。
- イベント参加者一覧は「誰が来ているか」の表示と、各参加者の**永続プロフィール閲覧**（ログイン時）までを担う。一覧から直接コネクションを張る導線は設けない。
- **保留中の招待（シナリオC）は削除しない**。これはオンライン接続ではなく、**対面でつながりQRをスキャンした**がアカウント未所持だった相手の接続を後から完成させる仕組みであり、「対面の握手」という不変条件を満たす。QR一本化の哲学と整合するため、未実装の優先機能として残す。

## 却下した代替案

- **チェックインに実在ゲートを追加（方針1）**: 取りこぼし救済というニッチな便益のために、GPS許可・屋内精度・実装コストという重い投資を払うことになる。便益に対して割高。
- **承認制で緩和してオンライン接続を作る（方針2）**: ADR-0007 が排除したスパム面を一部呼び戻す。コネクション自体は勝手に増えなくても「申請スパム」が残り、「コネクション＝対面で会った対称な記録」という意味が薄まる。

## トレードオフ

- **失うもの**: その場でQR交換しそびれた相手と、後からアプリ内で接続する手段。参加者一覧に「20人参加」と見えても、相手とつながるには結局その場でQRを出し合う（または相手のプロフィールを閲覧して各自のSNSへ飛ぶ）必要がある。
- **得るもの**: ADR-0007 のスパム排除と「コネクション＝対面で会った対称な記録」という意味を、実在検証という重い仕組みなしで維持できる。実装範囲が縮小し、優先実装（保留中の招待）に資源を集中できる。
- **許容できる理由**: シナリオAの便益は、実在ゲート化（重い投資）かスパム面の再導入（ADR-0007 の後退）のどちらかを払って初めて成立する程度のもので割に合わない。「同じ場にいた」を検証できない以上、参加者一覧からの接続は「会った人とつながる」ではなく「遠隔申請」に堕する。取りこぼしは対面QRの運用で受け止めるのが、プロダクトの背骨（対面の出会い）と最も整合する。
