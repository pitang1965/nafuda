# ADR-0020: チェックインは開催期間と QR 提示でゲートし、GPS 近接判定を持たない／即時イベントの文脈は交換時に確定する

- **ステータス**: 採用
- **決定日**: 2026-06-21
- **関連**: ADR-0012（即時イベント文脈の独立コピー）を補強。ADR-0013（チェックインへの GPS 実在ゲート却下）の方針を、別文脈（チェックイン受付ゲート）でも踏襲。

## 背景

「企画イベントのQR（`/e/{shareToken}`）はいつまで有効か」という問いから、チェックインの受付条件が未検討だったことが判明した。

- `checkinToEvent`（`event.ts`）は**時間検証を一切持たない**。`events.eventDate` は単一の日付で、表示用にしか使われていない。URL/QR を知っていれば**いつでも・無期限にチェックインできる**。
- つながる瞬間、`connection.ts` は**発行者Aのアクティブな（`checkedOutAt IS NULL`）チェックインを時間制限なしで読み**、それをコネクションの文脈としてスナップショットする。
- チェックアウトは「次のチェックイン時」にしか自動発生しない。

ここから2つの害が出る：

- **害①（名簿汚染）**: 後日 URL を拾った無関係者が「参加した」ことにでき、参加者一覧が嘘で埋まる。
- **害②（古いアクティブ状態の汚染）**: 2週間前のイベントのアクティブチェックインが残ると、今日のなふだ交換にそれが文脈として自動付与される。

Connpass のように受付期間や GPS 近接でゲートすべきか、という問いとして整理した。

## 問題

チェックインの目的は Connpass（出席証明・精算）とは異なり、(a) 参加者名簿、(b) コネクションの文脈付与の2つ。精算も本人確認も無いため、Connpass 並みの厳格な実在検証は過剰になりうる。一方で無期限チェックインは害①②を放置する。

論点は次の3つに分かれた：

1. チェックイン受付に時間窓を設けるか、設けるならどう定義するか（`eventDate` は単一点で終了時刻が無い）。
2. GPS 近接判定で「会場に居ること」を検証するか。
3. 即時イベントのアクティブチェックインの寿命をどう管理するか（開催期間を持たないため終了による失効が使えない）。

## 決定

**1. イベントに「開催期間 `[開始, 終了]`」を持たせ、チェックインをその範囲（前後に短い猶予）でゲートする。**

- 単一 `eventDate` を `[開始, 終了]` の範囲へ昇格する（日付のみの複数日開催も、日時の時間帯も表せる）。
- チェックイン可否はこの範囲＋前後の短い猶予で律する。範囲外（後日 URL を拾った等）からの参加は受け付けない → 害①が消える。
- 前側は実害が薄い（会場に居なければスキャンできない）ので緩く、後側は害①の本丸なので締める。猶予幅は「現場に実際に居た人だけ」を保つ短さ（前後それぞれ十数分程度）とし、定数で調整可能にする。

**2. GPS 近接判定は持たない。QR/URL の提示自体が"その場に居る人だけ"のゲートを兼ねる。**

- 会場にいる人にだけ QR を見せる＝Zoom に参加中の人にだけ URL を見せる。物理でもオンラインでも、QR 提示が同席ゲートになる。
- よって online/offline をイベント種別として区別しない。「どこで」が会場名でもオンラインの場でもよいだけ。オンラインイベント（Zoom 等）も同じ仕組みで対象に含む。

**3. 企画イベントは開催終了でアクティブチェックインを自動チェックアウトする。**

- 終了時刻が確定するため、終了をもって開きっぱなしのアクティブ状態を閉じられる → 害②が企画イベントで消える。範囲化して初めて可能になった仕掛け。

**4. 即時イベントの文脈は交換のその場で確定し、残存アクティブチェックインからは引かない。**

- 即時イベントは開催期間を持たない。そこで「Nh で自動失効」させる代わりに、**コネクション文脈の出どころを「その交換で作られた即時イベント」だけに限定**し、過去の残存アクティブチェックインを後続の交換で再利用しないことにした（脱結合）。これにより寿命の当て推量（Nh）自体が不要になる。
- 文脈を提供するのは常に**つながりQR を差し出す側（発行者A）**。即時イベントは**交換ごとに新規・独立**に作られ、束ねない（グルーピングは企画イベントの名簿が担う）。

## 却下した代替案

- **GPS 近接ゲート**: そもそも土台が無い——企画イベントは座標を保存しておらず（保存しても作成時の作成者位置＝自宅等になりうる）、通常チェックインは参加者 GPS を取得していない。加えて屋内 GPS は±数十m ぶれて正規参加者を弾く偽陰性を生み、詐称も可能で「セキュリティ」にならない。QR 提示が既に同席を担保しており、近接判定は冗長（ADR-0013 と同じ結論を別文脈で再確認）。
- **online/offline をイベント種別にする**: QR ゲートは物理でもオンラインでも同一に効くため、種別を分ける必要がない。「どこで」の自由度で吸収する。
- **開催期間と独立した受付期間（オンライン事前チェックイン）**: 「事前にチェックインした人だけに限定情報を提供する」という RSVP＋限定コンテンツは、対面の出会いという打ち出し（ADR-0009）の核から外れ、別機能を芋づるで連れてくる。さらに"チェックイン＝その場に居た"の語義を「事前の関心表明」で濁す。将来必要になれば別概念として導入する余地は、開催期間を範囲で持つことで残してある。
- **即時チェックインに Nh 自動失効を与える**: N は「集まりの長さ」の当て推量にしかならず、外せば害②（長すぎ）か文脈の無言欠落（短すぎ）を生む。脱結合（決定4）で N の必要性そのものを消す方が、当て推量の定数をプロダクトから無くせて健全。

## トレードオフ

- **得るもの**: 害①②を、GPS という重い仕組みも N という当て推量の定数も使わずに消せる。オフライン／オンラインを単一モデルで扱える。「チェックイン＝開催中のイベントにその時居た」「文脈の出どころは差し出す側A」という意味が締まる。
- **失うもの／代償**: (a) `eventDate` 単一→範囲のスキーマ移行と、既存イベントのマイグレーションが要る。(b) 即時イベントで同じ場の複数人と交換する際、文脈は交換ごとに確定する（黙って引き継がない）。同じ場を1つの即時イベントに束ねる「同じ即時イベントに関連する人々」の将来構想は、案アの採用により当面持たない。
- **許容できる理由**: なふだのチェックインには精算も本人確認も無く、Connpass 並みの厳格さは要らない。害の実体は「無期限の後追い」と「古いアクティブ状態の無言の汚染」で、どちらも開催期間ゲートと脱結合という軽い仕掛けで解ける。束ねない方が紙の名刺交換（1対1の繰り返し）のメタファーにも忠実。
