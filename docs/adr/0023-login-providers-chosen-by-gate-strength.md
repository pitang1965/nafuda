# ADR-0023: ログインプロバイダはゲート強度で選ぶ（X・Instagram はリンク専用、LINE を追加）

- **ステータス**: 採用
- **決定日**: 2026-06-17（番号採番: 当初 ADR-0016 として起票したが、staging で 0016 が CSRF ミドルウェアに割り当て済みだったため 2026-06-29 に 0023 へ採番し直した）

## 背景

「Xログインやメールログインも用意した方がいいのでは」という要望が出た。現状の認証（`src/server/auth.ts`、better-auth）は **Google + Facebook** の2プロバイダ。一方、メイン想定ユーザー「もちこ（23歳・推し活）」（`docs/02_needs/user-personas.md`）は X を主戦場とし、出会いの場面は「X のアカウント教えてください」である。

直感的には「核心客層が使う X でログインできた方が親切」に見える。しかし要望の主目的を詰めると **「Google/Facebook を持たない／使いたくない人の取りこぼしを減らす」**（間口）であり、同時にオーナーは **「作りたてでない実態のあるSNSで認証することによるスパム・量産耐性（Sybil耐性）」** を重視していた。メール認証は「メールアドレスを持つこと」しか証明せず捨てアドで量産できるため、ゲートとして弱いと判断された。

## 問題

「人気のSNS」と「ログインに向くSNS」を同じ基準で選ぶと判断を誤る。守りたい脅威は **偽プロフィールの量産・なりすまし（Sybil攻撃）** であり、これに効くのは認証プロバイダの **ゲート強度（アカウント量産のしにくさ）** である。客層人気はゲート強度とは無関係な別軸だ。

主要プロバイダをゲート強度で並べると：

| プロバイダ | 量産耐性 | 根拠 |
|---|---|---|
| LINE | ◎ | 作成に電話番号(SMS)必須。日本では捨て番号の調達が高コスト |
| Google | ○ | しばしば電話確認を要求（導入済み） |
| Apple | ○ | Apple デバイス＋Apple ID（端末/課金に紐付く） |
| X | ✕ | 電話番号なしで量産可能。ボット農場の代名詞 |
| Instagram | ✕ | 同様に量産容易。加えて後述のAPI事情でログイン自体が困難 |

つまり **X を「ログイン」に足すと、最も嫌う“実態のない量産アカウント”に最も広い裏口を開ける**ことになり、要望の主目的（量産耐性のあるゲート）に対して逆効果になる。X の魅力は「馴染みのある入口」という体験面であって、ゲート面ではない。

加えて技術事情：

- **X**: 「Sign in with X」(OAuth 2.0) はログイン自体は Free tier で可能だが、レートリミットが厳しく運用が脆い。本名の出る X 本アカを初対面に出したくないという同ペルソナのプライバシー懸念とも緊張する。
- **Instagram**: Meta が **2024年12月4日に Instagram Basic Display API を廃止**。代替の「Instagram API with Instagram Login」はビジネス/クリエイター向けで重く、消費者の汎用ログインには実質使えない。

なお、なふだには既に2つの層がある——**①ログイン（認証）**と**②SNSリンク（公開なふだに載せる外部サービスへのリンク、`sns_links`）**。「Xでつながりたい」という客層のニーズは ②（`sns_links` の `x`・`instagram` 等、対応済み）が満たす役割であり、①に持ち込む必要はない。

## 決定

**ログインプロバイダはゲート強度と客層で選ぶ。人気だが量産しやすい／APIが非対応のSNS（X・Instagram）はログインに足さず、SNSリンク層（②）に留める。**

- **ログイン構成**: Google（維持）＋ Facebook（維持）＋ **LINE（追加）**。
  - LINE は日本ユーザーをほぼ網羅し、電話番号ゲートで量産耐性が最も高く、Login チャンネルは無料。better-auth の標準プロバイダには無いため `genericOAuth` プラグインで実装する。
  - Facebook は推し活層には刺さらないが、サブペルソナ（ヤマダ・サトウ世代の一般イベント展開）と既存ユーザー保護のため当面維持。廃止はローンチ後の利用データで再考する。
- **X・Instagram はログインに追加しない**。`sns_links` の `x`・`instagram` で「そのSNSでつながる」体験を提供する現状を正とする。X ログインは、登録離脱データで「X が無いことに起因する離脱」が実証された場合に限り再検討する。
- **将来トリガーで追加するプロバイダ**:
  - **Apple** — ネイティブ iOS アプリを出す時（App Store 規約上ほぼ必須になる。Apple Developer Program $99/年が前提）。
  - **GitHub** — 技術イベント／勉強会（Connpass 層）展開フェーズ。ペルソナB/C向けの入口で、メイン客層は持たない。
  - **Discord** — 推し活コミュニティで人気だがゲートは弱め。必要が生じれば②から①への昇格を検討。

## 却下した代替案

- **X ログインを追加する**: 客層の馴染みは最強だが、ゲートが全プロバイダ中で最弱で、要望の主目的（量産耐性）に逆行する。API運用の脆さと「本アカを出したくない」プライバシー懸念も重なる。「Xでつながる」は SNSリンク層で既に満たせる。
- **メール認証（メール＋パスワード／マジックリンク）を追加する**: 誰でも持てる受け皿だが、捨てアドで量産でき Sybil 耐性が無い。「実態のあるSNSで認証する」という安心感の主旨に反する。
- **Instagram ログインを追加する**: 客層適合は高いが、Basic Display API 廃止により消費者ログインが実質不能。残る Graph API はビジネス向けで重く、アカウント量産耐性も弱い。
- **Facebook を即廃止して LINE と入れ替える**: 推し活層には不要だが、一般イベント展開で効くサブ客層と既存ユーザーを切ることになる。廃止は早計で、データを見てから判断する。

## トレードオフ

- **失うもの**: 「核心客層が日常使う X でそのままログインできる」という体験上の親しみ。Instagram ログインという客層適合の高い選択肢。LINE 実装は標準プロバイダではなく `genericOAuth` のため Google/Facebook より手間がかかる。
- **得るもの**: 量産・なりすましに対する強いゲート（LINE の電話番号ゲート＋Google）。「ログイン＝ゲート／人気SNS＝リンク層」という選定軸の明文化。将来「なぜ核心客層の X でログインできないのか」という当然の疑問への回答。X/Instagram の脆い・非対応なAPIへの依存回避。
- **許容できる理由**: 要望の主目的は「量産耐性のある間口を増やす」であって「人気SNSを入口にする」ことではなかった。人気は SNSリンク層（②）で活かせるため、ログイン層（①）はゲート強度で選ぶのが筋が通る。X/Instagram を入れて得る親しみより、量産耐性とAPIの安定を取る方がプロダクトの安心設計（同担拒否・本名を出さない出会い）と整合する。

## 補足: LINE アカウントの自動リンクを信頼する（2026-06-29）

better-auth の `line()` プラグイン（`genericOAuth`）は、プロフィールを id_token から取り出す際に保守的に **`emailVerified: false`** を返す。一方で `src/server/auth.ts` の `accountLinking.trustedProviders` には **`line` を含めている**。これにより、同一メールアドレスの既存アカウント（例: Google で登録済み）に対し、LINE ログインを**無検証で自動リンク**する（「Google → LINE で同じ人 ＝ 1 アカウント」）。

- **そう決めた根拠**:
  - LINE はアカウントへのメール登録時に **LINE 側で確認**しており、id_token で返るメールは実質検証済みとみなせる（`emailVerified:false` は汎用デフォルトに過ぎない）。
  - 本 ADR の主脅威は **Sybil（量産）** であり、メール衝突によるアカウント乗っ取りは casual な出会いアプリでは優先度が低い。
  - 本アプリは `emailVerified` を認可ゲートに使っている箇所が無い（`schema` の列のみ）。よって列が `false` でも実害がなく、自動リンクの UX（1 アカウント統合）を優先する方が利益が大きい。
- **再考トリガー**: メール衝突によるアカウント乗っ取りを脅威に格上げする場合は、`line` を `trustedProviders` から外し、リンク前に既存プロバイダでの再認証を要求する設計に切り替える。

なお、メール permission 通過後も **LINE アカウントにメール未登録のユーザー**、および**同意画面でメール提供を OFF にしたユーザー**は存在し得る。その場合 id_token にメールが無く `user.email`（NOT NULL）が埋まらずサインアップが失敗するため、ログイン画面では原因別のエラー文言（Google への誘導）で受け止める。

better-auth はこのケースで `mapProfileToUser` によるダミーメール合成（例 `<sub>@line.local`）も提示するが、**採らない**。合成メールは実在の Google メールと一致せず、上記のメール一致による自動リンクを壊して同一人物が二重アカウントになるうえ、本人識別子である `email` 列に配送不能の値を混ぜることになるため。メール未取得の LINE ユーザーはブロックし、Google ログインへ誘導する方針を正とする。
