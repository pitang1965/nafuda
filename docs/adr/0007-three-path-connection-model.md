# ADR-0007: 「つながる」の3経路設計とコネクションの双方向即時生成

- **ステータス**: 採用（経路③「イベントリクエスト」は ADR-0013 で撤回・削除）
- **決定日**: 2026-06-04

## 背景

当初の「つながる」はShareToken URLにアクセスできる誰でも可能だった。ユーザーが自分のなふだURLをSNSやWebサイトにlinktr.ee代わりに公開するユースケースが生まれたことで、見知らぬ人が勝手にコネクション一覧に追加されるスパム的問題が顕在化した。

また、旧設計ではBがタップするとB→Aの1行のみ生成されていた。Aはコネクション一覧でBを見られるが、Aは能動的に「つながった」わけではない非対称な状態だった。「名刺交換」という本来の意図とは齟齬があった。

## 決定

### 1. 「つながる」を3経路に分離する

| 経路 | 説明 | コネクション生成 |
|---|---|---|
| **永続プロフィールURL** | linktr.ee代わりの公開・印刷用。「なふだを見せる」 | 生成しない |
| **つながりQR（短命15分）** | その場での名刺交換。「なふだを交換する」 | 即時双方向 |
| **イベントリクエスト** | イベント参加者一覧からの申請・承認 | 承認後に双方向 |

永続URLからの「つながる」は廃止する。

### 2. コネクションを即時双方向・非対称削除とする

「なふだを交換する」とイベントリクエスト承認の両ケースで、A→BとB→Aの2行を同時に生成する。削除は非対称で、各自が自分の行を独立して削除できる（相手の行には影響しない）。

### 3. アカウント未作成者への保留招待

つながりQRをスキャンした時点でアカウントを持たないユーザーに対してのみ、サーバーが「保留中の招待」レコードを作成する（スキャンから48時間有効）。アカウントあり・ペルソナなしの場合はウィザードを同一セッション内で完結できるためDBレコードは不要。アカウントおよびペルソナ作成後に自動適用し、コネクションを双方向生成する。これにより「名刺を持っていない相手が後から送る」シナリオに対応する。

### 4. イベントリクエストの自動失効

イベントリクエストはイベント終了から14日で自動失効する。拒否ボタンは設けない（承認か無視のみ）。失効は双方に通知しない。

## 却下した代替案

- **フォロー概念の追加**: 「つながる」と「フォロー」で関係強度を分ける案。語彙と関係モデルが複雑になるため却下
- **URLパラメータで期限なし交換モード**: URLパラメータで「交換モード」を判別し永続的に有効にする案。期限切れが唯一の防壁であるため却下。URLが流出すると永続的に接続可能になる
- **公開URLに承認設定を追加**: ペルソナごとに「つながりを承認制にする」トグルを設ける案。ユーザーの設定負担と実装の複雑さが増すため却下

## トレードオフ

- **失うもの**: 永続URLからの「つながる」。知人がlinktreeとして公開されているURLを見てつながれなくなる
- **得るもの**: スパム接続の構造的排除。コネクションが「出会いの記録」として対称な意味を持つ
- **許容できる理由**: なふだの主要ユースケースはQRを使った対面交換であり、URLを見た人が後からつながるニーズは副次的。副次的ニーズのためにスパムリスクを許容する必要はない
